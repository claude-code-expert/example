Git 기본 개념 — 용어부터 정리하기

Git은 파일의 변경 이력을 저장하고 되돌릴 수 있게 해주는 버전 관리 도구입니다. “언제 · 누가 · 무엇을 · 왜” 바꿨는지 기록해 두기 때문에, 잘못돼도 과거로 돌아갈 수 있고 여러 명이 같은 프로젝트를 안전하게 함께 고칠 수 있습니다. 시작하기 전에 아래 용어만 알아두면 됩니다.

꼭 알아야 할 용어

용어한 줄 뜻
저장소 (Repository · repo)프로젝트 파일과 그 모든 변경 이력이 담긴 공간. git init 또는 git clone으로 만듭니다.
커밋 (Commit)그 순간의 변경 내용을 기록한 저장점(스냅샷). 메시지를 붙여 “무엇을 왜 바꿨는지” 남깁니다.
작업 디렉터리 (Working Directory)지금 내가 파일을 실제로 열어 고치고 있는 폴더.
스테이징 영역 (Staging Area)다음 커밋에 담을 변경만 골라 올려두는 중간 칸. (아래에서 자세히)
로컬 저장소 (Local)내 컴퓨터 안에 있는 저장소. 인터넷 없이도 커밋할 수 있습니다.
원격 저장소 (Remote · 예: GitHub)인터넷에 올려 팀과 공유하는 저장소. 보통 별칭 origin으로 부릅니다.
브랜치 (Branch)본류에 영향을 주지 않고 따로 작업하는 갈래. 기본 브랜치는 보통 main.
HEAD지금 내가 작업 중인 현재 위치를 가리키는 표시.
업스트림 (Upstream)내 로컬 브랜치가 따라가는 짝꿍 원격 브랜치(예: origin/main). git push -u로 한 번 맺어두면, 이후엔 git push·git pull만 쳐도 그 브랜치와 주고받습니다.
기본 흐름은 딱 세 글자

파일을 고친 뒤 add → commit → push 순서로 진행합니다. 고른 변경을 담고(add), 내 컴퓨터에 기록하고(commit), 원격에 올립니다(push). 이 가운데 처음 만나는 벽이 바로 add, 즉 스테이징입니다.

스테이징 — “택배 상자에 담기”로 이해하기

핵심 한 줄

스테이징은 “이번에 보낼 것만 골라 담는 선별 바구니”입니다. 파일 10개를 고쳤어도 관련된 3개만 git add 해서 따로 커밋할 수 있게 해줍니다.

단계 한눈에 보기

상태비유위치명령
수정만 함책상에 물건이 어질러짐작업 디렉터리—
담음택배 상자에 담기스테이징git add
확정봉인·송장 붙여 현관에 보관로컬 저장소git commit
발송택배기사가 가져감원격(GitHub)git push
주의 — commit ≠ 발송

발송(택배기사가 가져감)은 push입니다. commit은 “내 집(로컬)에 보관 확정”까지라서, commit만 하면 GitHub엔 아직 보이지 않습니다. push해야 올라갑니다.

가장 흔한 오해 두 가지

  • add는 파일을 복사·저장하는 게 아니다. “이번 커밋 후보로 표시”하는 것일 뿐입니다.
  • add 후 또 수정하면 다시 add 해야 한다. 상자에 담은 뒤 물건을 또 만졌으면 다시 담아야 하는 것과 같습니다.
상태는 git status로 확인

지금 어떤 파일이 스테이징됐고 어떤 파일이 아직 작업트리에만 있는지는 git status로 한눈에 볼 수 있습니다. staged = 상자에 담김, unstaged = 아직 책상 위로 표시됩니다.

기본 명령어 — 워크플로우 필수 한 줄 정리

협업 워크플로우에서 가장 자주 쓰는 명령들입니다. 자세한 흐름은 02 · 기본 워크플로우 탭에서 단계별로 확인하세요.

명령한 줄 설명
git clone <주소>원격 저장소 전체(이력 포함)를 내 컴퓨터로 복제해 처음 받아옵니다.
git status지금 변경·스테이징 상태와 현재 브랜치를 확인합니다.
git add <파일>변경한 파일을 다음 커밋 후보로 스테이징합니다 (git add .는 전체).
git commit -m "메시지"스테이징한 내용을 로컬에 커밋(기록)합니다.
git fetch원격의 최신 변경을 가져오기만 합니다 (내 작업에는 아직 안 합침 — 확인용).
git pull원격의 변경을 가져와 현재 브랜치에 바로 합칩니다 (fetch + merge).
git push내 로컬 커밋을 원격에 올립니다 (첫 푸시는 git push -u origin <브랜치>).
git branch브랜치 목록 확인 (git branch <이름>은 새 브랜치 생성).
git switch <브랜치>다른 브랜치로 이동합니다 (git switch -c <이름>은 만들면서 이동, 구버전 checkout).
git merge <브랜치>지정한 브랜치의 작업을 현재 브랜치로 합칩니다.
기억할 순서

혼자 시작: clone → add → commit → push. 협업 중: 작업 전 pull로 최신화 → 작업 → add·commit → push. 이 한 줄만 몸에 익혀도 대부분의 협업이 굴러갑니다.

이 개념을 잡았다면, 다음 02 · 기본 워크플로우 탭에서 add·commit·push가 로컬과 원격 사이에서 실제로 어떻게 움직이는지 단계별로 확인해 보세요.

기본 워크플로우 — 로컬 ↔ 원격

버튼으로 한 단계씩 진행하며 로컬 ↔ 원격, 브랜치 이동, push·pull·merge가 어떻게 일어나는지 눈으로 확인하세요.

색 규칙 — 앰버색은 “지금 단계에서 일어나는 동작/현재 위치(HEAD)”만 표시합니다. 회색 사각형은 커밋 1개, HEAD 는 현재 작업 중인 브랜치를 뜻합니다. git switch 는 구버전의 git checkout 에 해당합니다.

되돌리기 — 커밋 · 푸시 · 머지 · 소스

버튼으로 한 단계씩 진행하며 커밋·푸시·머지·소스를 어떻게 안전하게 되돌리는지 확인하세요.

주의 — 안전 원칙

reset --hard·--force는 아직 push 안 한 로컬에서만 쓰세요. 이미 원격에 올라갔고 협업 중이라면 이력을 보존하는 git revert가 정답입니다.

읽는 법 — 회색 사각형은 커밋 1개, 앰버 사각형은 “이번 단계에서 생긴/관련된 커밋”, 점선 사각형은 “되돌려져 사라진 커밋”입니다.

UI 명칭·버전·정책은 시점에 따라 달라질 수 있습니다. 명령은 git 2.43에서 실제 실행으로 검증했으며, 인증·토큰 관련 세부는 공식 문서에서 최신 내용을 확인하세요.
