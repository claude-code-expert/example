00ERD란? 왜 필요한가개념

ERD(Entity-Relationship Diagram)는 데이터베이스의 구조를 그림으로 나타낸 설계도입니다. DB를 이루는 엔터티(Entity, 표·테이블)와 엔터티 사이의 관계(Relationship, 연결)를 한눈에 보여 줍니다.

집 짓기 전 설계도가 필요한 이유

집을 지을 때 설계도 없이 바로 벽돌을 쌓으면 어떻게 될까요? 방 크기를 나중에 바꾸려면 벽을 허물어야 합니다. DB도 마찬가지입니다. 코드를 다 짜고 나서 "이 테이블에 컬럼이 하나 더 필요한데"를 깨달으면, 이미 쌓인 코드 전체를 고쳐야 합니다. ERD는 그 설계도입니다. 구현 전에 그림으로 합의해 두면, 수정 비용이 벽돌 대신 지우개 한 번으로 줄어듭니다.

ERD가 해결하는 세 가지 문제

  • 누락 조기 발견 — "주문에 배송 주소 컬럼이 없네"를 코딩 전에 잡는다.
  • 중복 제거 — 같은 데이터를 두 군데 저장해 정합성이 깨지는 상황을 예방한다.
  • 팀 공유 — 개발자·기획자·디자이너가 같은 그림을 보고 대화할 수 있다.
ERD는 "계획"이지 코드가 아닙니다

ERD를 그린다고 DB가 만들어지지 않습니다. ERD는 어떤 테이블이 필요하고 어떻게 연결되어야 하는지를 합의하는 도구입니다. 합의가 끝나면 그것을 토대로 SQL CREATE TABLE 문을 작성합니다.

01핵심 부품 — 엔터티·속성·PK·FK개념

세 가지 기본 단위

용어DB에서 부르는 이름엑셀 비유
엔터티 (Entity)테이블 (Table)시트 한 장
속성 (Attribute)컬럼 (Column)열 제목 (이름, 나이…)
레코드 (Record)행 (Row)실제 데이터 한 줄

PK — Primary Key (기본 키)

테이블에서 각 행을 유일하게 식별하는 컬럼입니다. 주민등록번호처럼, 두 행이 같은 PK를 가질 수 없습니다.

  • 고유 (Unique) — 같은 값이 두 번 들어올 수 없다.
  • NOT NULL — 반드시 값이 있어야 한다.
  • 불변 권장 — 한 번 정한 PK는 바꾸지 않는 것이 좋다.

예: user 테이블의 user_id. 이메일이나 이름 대신 숫자 ID를 쓰는 이유는, 이메일·이름은 변경될 수 있지만 ID는 불변이기 때문입니다.

FK — Foreign Key (외래 키)

다른 테이블의 PK를 가리키는 컬럼입니다. "이 행은 저 테이블의 저 행에 속한다"는 연결 고리입니다.

예: post 테이블의 user_id 컬럼은 user 테이블의 user_id (PK)를 가리킵니다. 이 덕분에 "이 게시글은 누가 썼는가"를 쉽게 조회할 수 있습니다.

PK / FK 비교

항목PK (기본 키)FK (외래 키)
역할자기 행을 유일하게 식별다른 테이블의 PK를 참조
NULL 허용불가허용 (선택적 관계일 때)
중복 허용불가허용 (여러 행이 같은 FK 가능)
테이블당 개수보통 1개여러 개 가능

02관계의 종류 — 1:1, 1:N, N:1, N:M개념

두 테이블 사이에 "A 한 건에 B 몇 건이 대응하는가"를 읽는 것이 관계(Relationship)입니다.

관계 유형읽는 법예시
1:1A 한 건 ↔ B 정확히 한 건 (또는 없음)회원 ↔ 회원 프로필 (확장 정보)
1:NA 한 건 ↔ B 여러 건회원 한 명 → 주문 여러 건
N:1A 여러 건 ↔ B 한 건주문 여러 건 → 회원 한 명
N:MA 여러 건 ↔ B 여러 건학생 여러 명 ↔ 강의 여러 개
1:N과 N:1은 보는 방향만 다릅니다

"회원(1) → 주문(N)"과 "주문(N) → 회원(1)"은 완전히 같은 관계입니다. 어느 테이블을 중심으로 보느냐에 따라 이름이 달라집니다. 회원 입장에서 보면 1:N, 주문 입장에서 보면 N:1입니다.

03카디널리티 표기 — Crow's Foot표기법

ERD 선 끝에 붙는 기호가 카디널리티(Cardinality)를 표현합니다. "이 관계에서 최소 몇 건, 최대 몇 건이 대응하는가"를 선 기호로 압축한 것입니다. 가장 널리 쓰이는 표기법이 Crow's Foot(까마귀발)입니다.

세 가지 기본 기호

기호 이름의미값
동그라미 (Circle)없어도 됨 — 선택(Optional)0
작대기 (Bar)반드시 있어야 함 — 필수(Mandatory) / 정확히 하나1
까마귀발 (Crow's Foot)여러 건 가능 — 다수(Many)N

기호는 선 끝에 두 개가 세트입니다. 엔터티 바로 옆(바깥) 기호가 최소를, 선 중앙 쪽(안쪽) 기호가 최대를 나타냅니다.

왜 0과 1을 구분하나

0과 1의 차이는 "이 FK 컬럼이 NULL을 허용하는가"와 직결됩니다.

  • 최소 0 → FK 컬럼에 NULL 허용 (선택적 관계, LEFT JOIN 필요)
  • 최소 1 → FK 컬럼에 NOT NULL 제약 (필수 관계, INNER JOIN으로 충분)

ERD에서 작은 원 하나가 실제 DB 제약 조건(NOT NULL)과 API 입력 검증(필수 입력 여부)에 영향을 미칩니다.

04예제 ① 1:1 관계예제

1:1 관계는 A 한 건이 B 정확히 한 건(또는 없음)에 대응하는 관계입니다.

예: member ↔ member_profile

member는 로그인에 반드시 필요한 핵심 정보(이메일, 비밀번호)를 담고, member_profile은 닉네임·자기소개·프로필 이미지처럼 선택적이거나 자주 조회하지 않는 확장 정보를 담습니다.

왜 1:1로 분리하나

  • 성능 — 로그인 시에는 핵심 정보만 빠르게 조회합니다. 프로필은 필요할 때만 조인합니다.
  • 보안 — 민감한 컬럼을 별도 테이블로 격리해 접근 권한을 다르게 설정할 수 있습니다.
  • 선택적 존재 — 프로필을 작성하지 않은 회원이 있을 수 있어, member_profile 행이 없는 경우도 허용합니다.

CREATE TABLE 예제

-- 회원 핵심 정보
CREATE TABLE member (
  member_id  BIGINT       NOT NULL AUTO_INCREMENT,
  email      VARCHAR(255) NOT NULL UNIQUE,
  password   VARCHAR(255) NOT NULL,
  created_at DATETIME     NOT NULL DEFAULT CURRENT_TIMESTAMP,
  PRIMARY KEY (member_id)
);

-- 회원 확장 정보 (선택 — 없어도 됨)
CREATE TABLE member_profile (
  member_id  BIGINT        NOT NULL,
  nickname   VARCHAR(50),
  bio        TEXT,
  avatar_url VARCHAR(500),
  updated_at DATETIME      NOT NULL DEFAULT CURRENT_TIMESTAMP,
  PRIMARY KEY (member_id),
  CONSTRAINT fk_profile_member
    FOREIGN KEY (member_id) REFERENCES member(member_id)
    ON DELETE CASCADE
);
ON DELETE CASCADE 란?

member 행을 삭제하면 연결된 member_profile 행도 자동으로 삭제됩니다. 부모가 사라지면 자식도 함께 사라지는 설정입니다. 반대로 ON DELETE RESTRICT는 자식 행이 있으면 부모 삭제를 막습니다.

05예제 ② 1:N / N:1 관계예제

가장 흔한 관계입니다. "1쪽" 테이블의 PK를 "N쪽" 테이블이 FK로 가져갑니다.

예: member(1) ─── order(N)

회원 한 명이 여러 번 주문할 수 있습니다. 반대로 주문 한 건은 반드시 회원 한 명에게 속합니다. order 테이블이 member_id FK를 가집니다 — "이 주문이 어느 회원의 것인지"를 기록하기 위해서입니다.

FK는 언제나 N쪽 테이블에 있습니다

order 입장에서 보면 "주문 여러 건(N) → 회원 한 명(1)"이므로 N:1입니다. FK는 언제나 N쪽 테이블에 있습니다. 1쪽(member)은 FK를 가지지 않습니다. "어느 회원의 주문인가"를 주문 행에 직접 적어 두는 것이 더 자연스럽기 때문입니다.

SQL 예제

CREATE TABLE `order` (
  order_id     BIGINT         NOT NULL AUTO_INCREMENT,
  member_id    BIGINT         NOT NULL,        -- FK: 어느 회원의 주문인가
  total_amount DECIMAL(12, 2) NOT NULL,
  status       VARCHAR(20)    NOT NULL DEFAULT 'PENDING',
  ordered_at   DATETIME       NOT NULL DEFAULT CURRENT_TIMESTAMP,
  PRIMARY KEY (order_id),
  CONSTRAINT fk_order_member
    FOREIGN KEY (member_id) REFERENCES member(member_id)
);

-- 특정 회원의 주문 목록 조회
SELECT o.order_id, o.total_amount, o.ordered_at
FROM `order` o
WHERE o.member_id = 42
ORDER BY o.ordered_at DESC;

06예제 ③ N:M 관계예제

학생은 여러 강의를 수강하고, 강의는 여러 학생을 가집니다. 이것이 N:M 관계입니다.

N:M은 DB에 직접 만들 수 없다

관계형 DB는 N:M을 직접 표현하지 못합니다. 중간(연결) 테이블을 두어 1:N + N:1 두 개로 분해해야 합니다. 이 연결 테이블을 Junction Table, Bridge Table, Associative Table 등으로 부릅니다.

연결 테이블의 복합 PK

enrollment의 PK는 (student_id, course_id) 두 컬럼의 조합입니다. "한 학생이 같은 강의를 중복 수강할 수 없다"는 제약을 DB 레벨에서 보장합니다. 추가로 grade, enrolled_at 같은 관계 자체에 속하는 정보를 연결 테이블에 함께 저장할 수 있습니다.

SQL 예제

CREATE TABLE student (
  student_id  BIGINT       NOT NULL AUTO_INCREMENT,
  name        VARCHAR(100) NOT NULL,
  email       VARCHAR(255) NOT NULL UNIQUE,
  PRIMARY KEY (student_id)
);

CREATE TABLE course (
  course_id  BIGINT       NOT NULL AUTO_INCREMENT,
  title      VARCHAR(255) NOT NULL,
  instructor VARCHAR(100) NOT NULL,
  credits    TINYINT      NOT NULL DEFAULT 3,
  PRIMARY KEY (course_id)
);

-- 연결(Junction) 테이블 — N:M 분해
CREATE TABLE enrollment (
  student_id  BIGINT   NOT NULL,
  course_id   BIGINT   NOT NULL,
  grade       CHAR(2),
  enrolled_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
  PRIMARY KEY (student_id, course_id),          -- 복합 PK
  CONSTRAINT fk_enr_student
    FOREIGN KEY (student_id) REFERENCES student(student_id),
  CONSTRAINT fk_enr_course
    FOREIGN KEY (course_id)  REFERENCES course(course_id)
);

-- 특정 학생이 수강 중인 강의 목록 조회
SELECT c.title, c.instructor, e.grade
FROM enrollment e
JOIN course c ON e.course_id = c.course_id
WHERE e.student_id = 7;

ERD 완성도 체크리스트

엔터티

  • 모든 테이블에 PK가 있는가?
  • PK는 고유(Unique) · NOT NULL · 불변 값인가?
  • 같은 데이터가 두 테이블에 중복 저장되지 않는가?

관계

  • 모든 관계에 카디널리티(최소·최대)가 표기되어 있는가?
  • N:M 관계는 반드시 연결 테이블로 분해했는가?
  • FK는 참조하는 PK와 데이터 타입이 일치하는가?

NULL 및 제약

  • 최소 카디널리티가 0인 FK 컬럼에 NULL을 허용했는가?
  • 필수 입력 컬럼에 NOT NULL이 붙어 있는가?
  • 삭제 시 연결 데이터 처리 방침(ON DELETE CASCADE / RESTRICT)을 결정했는가?
