# Opus 4.8을 Fable 5처럼 운용하는 지침

> Claude Opus 4.8(`claude-opus-4-8`)로 분석 → 설계 → 개발을 진행할 때,
> Claude Fable 5(`claude-fable-5`)와 동일한 절차와 최대한 근접한 성능을 유지하기 위한 세부지침.
> Claude Code 세션(CLAUDE.md)과 API 직접 호출 양쪽에 모두 적용된다.

---

## 1. 두 모델의 차이 — 무엇을 보정해야 하는가

| 항목 | Fable 5 | Opus 4.8 | 보정 방법 |
|---|---|---|---|
| Thinking | **항상 켜짐** (파라미터 생략) | 생략 시 **꺼짐** | 모든 요청에 `thinking: {type: "adaptive"}` 명시 (§2) |
| Effort 기본값 | high (max까지 지원) | high (xhigh/max 지원) | 코딩·에이전트 작업은 `xhigh`로 상향 (§2) |
| 긴 자율 실행 | 기본기 (수 분 단위 턴) | 잘하지만 명세 의존도 높음 | 첫 턴에 전체 명세 투입 (§3) |
| 서브에이전트 활용 | 적극적, 비동기 병렬 능숙 | **보수적 — 잘 안 씀** | 위임 트리거를 프롬프트에 명시 (§4) |
| 검색·메모리·커스텀 툴 | 필요 시 알아서 사용 | **확신 있을 때만 사용** | "언제 쓰는지"를 툴 설명·프롬프트에 명시 (§4, §7) |
| 자율성 | 되돌릴 수 있으면 진행 | 사소한 결정도 **물어봄** | 소소한 결정은 스스로 하도록 지침 추가 (§7) |
| 자기 검증 | 스스로 검증 루프 구성 | 지시하면 잘함 | 검증 단계를 절차에 고정 (§5, §6) |
| 토크나이저 | Opus 4.8과 동일 | — | 보정 불필요 (토큰 수 동일) |
| 컨텍스트/출력 | 1M / 128K | 1M / 128K | 보정 불필요 |
| 가격 (1M 토큰) | $10 / $50 | $5 / $25 | Opus가 절반 — 검증·병렬을 더 돌려도 이득 |

핵심 결론: **격차의 대부분은 모델 지능이 아니라 기본 동작 성향 차이**다.
(1) thinking 명시, (2) effort 상향, (3) 명세 선행 투입, (4) 위임·검증 강제 — 이 네 가지로 대부분 메꿔진다.

## 2. 필수 설정 — 모든 요청 공통

### API 직접 호출 시

```python
response = client.messages.create(
    model="claude-opus-4-8",
    max_tokens=64000,                       # xhigh/max effort에서는 64K 이상
    thinking={"type": "adaptive"},          # 필수. 생략하면 thinking이 꺼져 Fable 대비 성능 급락
    output_config={"effort": "xhigh"},      # 코딩·에이전트: xhigh / 일반 분석: high / 기계적 작업: low
    messages=[...],
)
```

- `thinking` 생략 금지. Fable 5는 항상 켜져 있지만 Opus 4.8은 생략 시 꺼진다. 이 한 줄이 가장 큰 성능 차이를 만든다.
- `temperature` / `top_p` / `top_k` / `budget_tokens`는 Opus 4.8에서 400 에러 — 사용하지 않는다.
- 16K 초과 출력은 스트리밍(`messages.stream()` + `get_final_message()`).
- 긴 에이전트 루프는 Task Budget으로 자기 조절시킨다(베타 헤더 `task-budgets-2026-03-13`, 최소 20,000 토큰):

```python
output_config={"effort": "xhigh", "task_budget": {"type": "tokens", "total": 128000}}
```

### Claude Code 세션 시

- `/model opus` 로 Opus 4.8 선택. effort는 기본 xhigh — 그대로 둔다.
- §7의 시스템 프롬프트 지침을 프로젝트 CLAUDE.md에 넣는다.

## 3. 작업 단계 — Fable 5와 동일한 절차

Fable 5의 최대 강점은 "잘 명세된 목표를 받아 길게 자율 실행"이다. Opus 4.8도 같은 절차를 따르되,
**각 단계 진입·이탈 조건을 명시적으로 고정**해서 모델 재량에 맡기는 범위를 줄인다.

```
0. 명세  →  1. 분석  →  2. 설계  →  3. 구현  →  4. 검증  →  5. 회고/기록
```

| 단계 | 산출물 | effort | 이탈 조건 (다음 단계 진입 기준) |
|---|---|---|---|
| 0. 명세 | 목표·제약·완료 기준 1문서 | — (사람) | "완료"를 검증 가능한 문장으로 쓸 수 있음 |
| 1. 분석 | 영향 범위 맵 (파일:라인 목록, 데이터 흐름) | high | 변경이 닿는 모든 파일·호출자를 열거함 |
| 2. 설계 | 구현 계획 (단계별 변경 목록 + 트레이드오프) | xhigh | 각 변경이 파일 단위로 특정되고 리스크가 명시됨 |
| 3. 구현 | 코드 + 커밋 단위 | xhigh | 계획의 모든 항목 완료, 빌드/린트 통과 |
| 4. 검증 | 테스트 결과 + 실제 실행 확인 | xhigh | 완료 기준 전 항목이 도구 출력으로 입증됨 |
| 5. 기록 | 배운 것 1건 = 1파일 메모 | low | — |

**단계 0이 성패를 가른다.** Fable 5는 모호한 요청도 알아서 범위를 잡지만, Opus 4.8은 모호하면
중간에 되묻거나 좁게 해석한다. 첫 턴에 다음을 모두 넣는다:

```
목표: [무엇을, 왜 — 누구를 위해]
제약: [건드리면 안 되는 것, 기술 스택, 스타일]
완료 기준: [검증 가능한 문장. 예: "GET /users가 200과 함께 페이지네이션된 목록을 반환하고 테스트 X, Y가 통과한다"]
진행 방식: 분석 → 설계 → 구현 → 검증 순서로 진행하고, 각 단계 산출물을 보인 뒤 다음으로 넘어가라.
```

## 4. 업무 분배 — 메인 루프와 서브에이전트

Fable 5는 병렬 서브에이전트 위임이 기본기다. Opus 4.8은 **시키지 않으면 혼자 순차 작업**하므로
분배 규칙을 지침으로 고정한다.

### 분배 원칙

| 작업 성격 | 담당 | effort |
|---|---|---|
| 계획 수립, 설계 판단, 코드 작성, 최종 종합 | 메인 루프 (Opus 4.8) | xhigh |
| 코드베이스 탐색, 다중 파일 읽기, 사용처 조사 | 탐색 서브에이전트 (병렬) | low–medium |
| 독립 파일 단위 기계적 수정 (rename, 포맷 등) | 작업 서브에이전트 (병렬) | low |
| 구현 결과 리뷰, 반박형 검증, 테스트 결과 판정 | 검증 서브에이전트 (fresh context) | high–xhigh |

### 위임 트리거 (프롬프트에 그대로 사용)

```
작업이 독립적인 항목들로 갈라질 때(여러 파일 읽기, 여러 테스트 실행, 여러 후보 조사)는
순차로 처리하지 말고 같은 턴에 서브에이전트를 병렬로 띄워 위임하라.
반대로, 한 번의 응답으로 끝나는 작업(눈앞의 함수 리팩터링, 단일 파일 읽기)에는
서브에이전트를 만들지 마라.
검증은 구현한 컨텍스트와 분리된 새 서브에이전트에게 맡겨라 — 자기 검증보다 정확하다.
```

### 컨텍스트 관리

- 서브에이전트에는 **결론만 돌려받는다** — 파일 전문 덤프 금지. 메인 컨텍스트를 아낀다.
- 세션이 길어지면 단계 경계(설계 완료, 구현 완료)에서 산출물을 파일로 저장 — 컴팩션이 일어나도 절차가 유지된다.

## 5. 분석·설계 워크플로우 세부지침

### 5.1 분석 (단계 1)

1. **읽기 전에 목표 재진술**: 요구사항을 한 문단으로 재진술하고 사람 확인을 받는다 (해석 오류를 여기서 차단).
2. **병렬 탐색**: 관련 서브시스템별로 탐색 서브에이전트를 동시에 띄운다 — 진입점, 데이터 모델, 기존 유사 구현, 테스트 현황.
3. **영향 범위 맵 작성**: 변경이 닿는 파일을 `파일:라인` 단위로 열거. "이 함수의 호출자 전부"까지 추적한다.
   Opus 4.8은 확신 없으면 검색을 덜 하므로, **"관련 파일을 전부 열거할 때까지 탐색을 계속하라"고 명시**한다.
4. **모르는 것 목록화**: 가정으로 남는 항목을 명시적으로 적는다. 설계 단계에서 하나씩 해소.

### 5.2 설계 (단계 2)

1. **대안 2–3개 강제**: Opus 4.8은 첫 설계에 안착하는 경향이 있다. "접근법 2~3개를 제시하고 트레이드오프를 비교한 뒤 하나를 추천하라"를 지침에 넣는다.
2. **설계 산출물 형식 고정**:
   ```
   ## 변경 목록
   - [파일 경로]: [무엇을 어떻게] — [이유]
   ## 순서와 의존성
   ## 리스크와 롤백 방법
   ## 완료 기준과의 대응 (기준 항목 ↔ 변경 항목 매핑)
   ```
3. **설계 리뷰를 검증 서브에이전트로**: 설계안을 "이 설계를 반박하라, 빠진 엣지 케이스를 찾아라" 프롬프트의
   별도 에이전트에 넣는다. Fable 5의 self-verification을 외부화하는 것.
4. **사람 승인 게이트**: 설계 확정 전 구현 시작 금지. (Claude Code에서는 Plan Mode 활용.)

## 6. 개발 워크플로우 세부지침

### 6.1 구현 (단계 3)

1. **계획 항목 단위로 진행**: 설계의 변경 목록을 위에서부터 하나씩. 항목 완료마다 빌드·린트.
2. **범위 이탈 금지 지침** (Fable 5 마이그레이션 가이드의 no-tidying 지침을 그대로 사용):
   ```
   작업이 요구하는 것 이상의 기능 추가, 리팩터링, 추상화 도입을 하지 마라.
   버그 수정에 주변 정리를 끼워 넣지 마라. 가정 불가능한 시나리오에 대한
   에러 처리·폴백·검증을 추가하지 마라. 시스템 경계(사용자 입력, 외부 API)에서만 검증하라.
   ```
3. **진행 보고는 증거 기반** (Fable 5의 grounded-progress 지침):
   ```
   진행 상황을 보고하기 전에 각 주장을 이 세션의 도구 출력과 대조하라.
   증거를 가리킬 수 있는 작업만 완료로 보고하고, 미검증 항목은 미검증이라고 말하라.
   테스트가 실패하면 출력과 함께 실패했다고 보고하라.
   ```
4. **커밋 규율**: 논리 단위 1개 = 커밋 1개. 구조 변경과 동작 변경을 섞지 않는다.

### 6.2 검증 (단계 4)

1. **완료 기준 역추적**: 단계 0의 완료 기준을 체크리스트로 펼치고, 항목마다 "어떤 명령의 어떤 출력이 이를 입증하는가"를 채운다.
2. **테스트 + 실제 실행**: 테스트 통과만으로 끝내지 않는다 — 변경된 흐름을 실제로 한 번 구동한다 (앱 실행, API 호출, CLI 실행).
3. **Fresh-context 리뷰**: 구현 diff를 새 검증 에이전트에 넣고 리뷰시킨다. 이때 반드시:
   ```
   발견한 문제를 확신이 낮거나 사소해 보여도 전부 보고하라. 중요도나 확신으로
   거르지 마라 — 필터링은 다음 단계에서 한다. 각 발견에 확신도와 심각도를 붙여라.
   ```
   Opus 4.8은 "심각한 것만 보고하라"는 지시를 문자 그대로 따라서 recall이 떨어진다. 커버리지 우선으로 시키고 뒤에서 거른다.
4. **실패 시**: 증상이 아니라 근본 원인을 고친다. 수정 후 검증 전체를 다시 돈다.

### 6.3 기록 (단계 5)

교훈 1건 = 파일 1개, 맨 위에 한 줄 요약. 이미 저장소·이력에 있는 내용은 저장하지 않고,
기존 메모와 겹치면 새로 만들지 말고 갱신한다. 다음 세션 시작 시 이 메모를 먼저 읽도록 지침에 포함한다.
(Opus 4.8은 메모리를 잘 안 쓰므로 "몇 턴 이상 걸리는 작업 전에는 메모 파일을 확인하라"를 명시.)

## 7. 시스템 프롬프트 / CLAUDE.md 스니펫 (복붙용)

아래 블록을 CLAUDE.md 또는 API `system`에 넣으면 §1 표의 성향 격차가 대부분 보정된다.

```markdown
## 작업 방식 (Opus 4.8 → Fable 5 패리티)

### 절차
분석 → 설계 → 구현 → 검증 순서를 지킨다. 각 단계의 산출물(영향 범위 맵, 변경 목록,
diff, 검증 결과)을 제시한 뒤 다음 단계로 넘어간다. 설계 확정 전에는 코드를 수정하지 않는다.

### 자율성
사소한 결정(이름, 포맷, 기본값, 동등한 접근법 중 선택)은 합리적인 것을 골라 진행하고
선택 사실만 기록하라 — 묻지 마라. 범위 변경이나 파괴적 작업만 먼저 확인하라.
행동할 정보가 모이면 즉시 행동하라. 이미 확정된 사실을 다시 도출하거나,
선택하지 않을 대안을 나열하지 마라.

### 위임
독립 항목으로 갈라지는 작업(다중 파일 읽기, 다중 테스트, 다중 후보 조사)은
같은 턴에 서브에이전트를 병렬로 띄워 위임하라. 한 응답으로 끝나는 작업에는 만들지 마라.
검증은 구현과 분리된 새 에이전트에 맡겨라.

### 탐색과 도구
답이 대화에 없는 정보에 의존하면 기억으로 답하지 말고 먼저 검색·조회하라.
몇 턴 이상 걸리는 작업 전에는 메모 파일에서 관련 맥락을 확인하고, 새로 알게 된 것을 기록하라.

### 범위
작업이 요구하는 것 이상의 기능·리팩터링·추상화를 추가하지 마라. 가정 불가능한
시나리오의 에러 처리를 넣지 마라. 시스템 경계에서만 검증하라.
```
