# 3.5.2 AI에게 더 나은 결과를 이끌어내는 프롬프팅 기법

AI 코딩 도구에게 "회원가입 기능 만들어줘"라고 말하면 코드는 나온다. 하지만 원하는 코드가 나올지는 별개의 문제다. 같은 요청이라도 어떻게 전달하느냐에 따라 결과의 품질이 크게 달라진다. 이 절에서는 프롬프트 엔지니어링의 핵심 기법들을 소개하고, Tika 칸반 보드 프로젝트의 실제 개발 상황에 적용하는 방법을 살펴본다.

## 프롬프트 엔지니어링이란

프롬프트 엔지니어링(Prompt Engineering)은 AI에게 전달하는 지시문(프롬프트)을 설계하여 원하는 결과를 이끌어내는 기술이다. 단순히 "잘 부탁하는 것"이 아니라, AI 모델의 동작 원리를 이해하고 그에 맞게 입력을 구조화하는 체계적인 접근 방식이다.

AI 코딩 도구와의 협업에서 프롬프트 엔지니어링이 중요한 이유는 명확하다. AI는 모호한 지시에 모호한 결과를 내놓는다. 반대로 구조화된 지시에는 구조화된 결과를 내놓는다. 개발자가 코드의 품질을 통제하려면, 코드를 직접 작성하는 것만큼이나 프롬프트를 정교하게 작성하는 데 신경 써야 한다.

다음은 AI 코딩 도구 사용에 특히 유용한 6가지 프롬프팅 기법이다.

---

## 1. 제로샷 프롬프팅 (Zero-Shot Prompting)

### 개념

제로샷 프롬프팅은 예시나 사전 학습 없이, 지시문만으로 AI에게 작업을 요청하는 가장 기본적인 방식이다. AI 모델이 사전 학습(Pre-training)으로 이미 보유한 지식에 의존하여 응답을 생성한다.

"제로샷(Zero-Shot)"이라는 이름은 기계 학습에서 유래한다. 모델이 특정 작업의 예시를 한 번도(Zero) 보지 않고도(Shot) 수행한다는 의미다.

### 언제 사용하는가

- 작업이 단순하고 명확할 때
- 기대하는 결과의 형태가 분명할 때
- 별도의 컨텍스트나 예시가 필요 없을 때
- CRUD 같은 표준적인 패턴의 작업일 때

### 핵심 원칙

제로샷이 효과적이려면 지시문 자체가 충분히 구체적이어야 한다. "적당히 만들어줘"가 아니라, 무엇을(What), 어디서(Where), 어떻게(How)를 명시한다. 지시가 구체적일수록 AI가 추측해야 할 부분이 줄어들고, 원하는 결과에 가까운 코드가 나온다.

### Tika 앱 적용 예시

**예시 1: 데이터 조회 쿼리**

```
tickets 테이블에서 status가 'DONE'이 아니고 
due_date가 오늘 이전인 티켓을 조회하는 
Drizzle ORM 쿼리를 작성해줘.
```

이 프롬프트가 효과적인 이유는 대상 테이블(tickets), 조건(status ≠ DONE, due_date < today), 사용 기술(Drizzle ORM)이 모두 명시되어 있기 때문이다. AI가 추측해야 할 부분이 거의 없다.

**예시 2: 컴포넌트 수정**

```
TicketCard 컴포넌트에서 priority가 HIGH일 때 
빨간색 뱃지를 표시하도록 수정해줘.
Tailwind CSS 클래스를 사용하고, 
MEDIUM은 파란색, LOW는 회색으로 구분해줘.
```

**예시 3: 유효성 검증 추가**

```
POST /api/tickets 엔드포인트에 Zod 유효성 검증을 추가해줘.
- title: 필수, 1~200자, 공백만 불가
- priority: LOW, MEDIUM, HIGH 중 하나, 기본값 MEDIUM
- dueDate: 선택, YYYY-MM-DD 형식, 오늘 이후 날짜만 허용
```

### 제로샷의 한계

제로샷은 AI가 "관례적으로 이렇게 한다"라고 판단할 수 있는 범위에서만 잘 동작한다. 프로젝트 고유의 코딩 컨벤션, 에러 메시지 형식, 응답 구조 같은 것은 관례가 아니라 약속이다. 이런 프로젝트 고유의 패턴을 따르게 하려면 퓨샷 프롬프팅이 필요하다.

## 2. 퓨샷 프롬프팅 (Few-Shot Prompting)

### 개념

퓨샷 프롬프팅은 원하는 입출력 패턴의 예시를 1~3개 제시하고, AI가 그 패턴을 학습하여 동일한 형태로 결과를 생성하도록 유도하는 기법이다.

"퓨샷(Few-Shot)"은 소수의(Few) 예시(Shot)만으로 패턴을 파악한다는 뜻이다. 예시가 1개면 원샷(One-Shot), 2~3개면 퓨샷이라 부른다.

### 언제 사용하는가

- 프로젝트 고유의 코딩 스타일이나 패턴을 따르게 할 때
- 반복적인 구조의 코드를 일관되게 생성할 때
- 에러 응답, 로깅, 테스트 등 팀의 약속된 형식이 있을 때
- "이런 식으로 해줘"라고 말로 설명하기보다 보여주는 게 빠를 때

### 핵심 원칙

퓨샷의 핵심은 예시의 품질이다. 예시가 일관된 패턴을 보여줘야 AI가 그 패턴을 올바르게 추출한다. 예시끼리 형식이 다르면 AI는 어떤 패턴을 따라야 할지 혼란스러워한다. 예시는 2~3개가 적당하다. 1개로는 패턴을 잡기 어렵고, 4개 이상이면 컨텍스트만 낭비된다.

### Tika 앱 적용 예시

**예시 1: API 에러 응답 패턴 통일**

```
다음 패턴으로 나머지 API 에러 핸들러를 만들어줘.

[예시 1 - 404 Not Found]
if (!ticket) {
  return NextResponse.json(
    { error: '티켓을 찾을 수 없습니다.' },
    { status: 404 }
  );
}

[예시 2 - 400 Bad Request]
if (!parsed.success) {
  return NextResponse.json(
    { error: '유효하지 않은 입력입니다.', details: parsed.error.flatten() },
    { status: 400 }
  );
}

이 패턴을 따라서 다음 케이스를 만들어줘:
- 서버 내부 오류 (500)
- 중복 티켓 제목 (409)
```

AI는 두 예시에서 공통 구조를 추출한다: `NextResponse.json`으로 응답, `error` 필드에 한국어 메시지, HTTP 상태 코드 명시. 이 패턴을 따라 500과 409 케이스를 생성한다.

**예시 2: 테스트 코드 스타일 통일**

```
테스트 코드를 다음 패턴으로 작성해줘.

[예시]
it('should create a ticket with default priority MEDIUM', async () => {
  const res = await POST('/api/tickets', {
    body: { title: '테스트 티켓' },
  });
  expect(res.status).toBe(201);
  expect(res.body.priority).toBe('MEDIUM');
});

이 패턴으로 다음 테스트 케이스를 추가해줘:
- title 없이 요청하면 400 반환
- title이 200자 초과면 400 반환  
- dueDate가 과거 날짜면 400 반환
- 생성된 티켓의 status는 항상 BACKLOG
```

예시 하나를 보여줬을 뿐인데 AI는 테스트 설명 형식(should~), 헬퍼 함수 사용(POST), 검증 방식(expect)까지 동일하게 맞춘다.

**예시 3: 커밋 메시지 형식 지정**

```
커밋 메시지를 다음 형식으로 작성해줘.

[예시 1] feat(ticket): 티켓 생성 API 엔드포인트 추가
[예시 2] fix(board): 드래그앤드롭 시 position 계산 오류 수정
[예시 3] test(ticket): 오버듀 판정 로직 단위 테스트 추가

지금 변경사항에 맞는 커밋 메시지를 작성해줘.
```

### 퓨샷과 CLAUDE.md의 관계

퓨샷 프롬프팅을 매번 반복하는 것은 비효율적이다. 프로젝트 전반에 적용되는 패턴이라면 CLAUDE.md에 한 번 정리해두는 것이 낫다. 세션이 시작될 때마다 자동으로 로드되므로, 매번 예시를 붙여넣지 않아도 AI가 해당 패턴을 따른다.

```markdown
# CLAUDE.md

## API 응답 형식
- 성공: { data: ... }
- 실패: { error: '한국어 메시지', details?: ... }
- HTTP 상태 코드를 반드시 명시할 것

## 테스트 형식
- describe: 대상 함수 또는 엔드포인트
- it: 'should'로 시작하는 영문 설명
- 한국어 테스트 데이터 사용 (예: '프로젝트 요구사항 정리')
```

퓨샷은 일회성 패턴 전달에, CLAUDE.md는 프로젝트 수준의 영구 패턴 전달에 사용한다. 둘은 대립하는 것이 아니라 보완 관계다.

## 3. 생각의 사슬 (Chain-of-Thought, CoT)

### 개념

생각의 사슬은 AI에게 최종 답을 바로 요구하는 대신, 중간 추론 과정을 단계별로 거치도록 유도하는 기법이다. 2022년 구글 연구팀이 발표한 논문에서 제시된 이후, 복잡한 추론 작업에서 AI의 정확도를 크게 향상시키는 핵심 기법으로 자리 잡았다.

핵심 아이디어는 간단하다. 사람도 복잡한 문제를 풀 때 머릿속에서 단계를 밟는다. AI도 마찬가지로, "단계별로 생각해봐(Think step by step)"라고 지시하면 중간 논리를 명시적으로 전개하면서 더 정확한 결론에 도달한다.

### 언제 사용하는가

- 여러 조건이 얽힌 비즈니스 로직을 구현할 때
- 설계 결정이 필요한 상황에서 근거를 함께 도출할 때
- 버그의 원인을 추적할 때
- "왜 이렇게 해야 하는가"의 근거가 코드만큼 중요할 때

### 핵심 원칙

생각의 사슬이 효과적이려면 단계의 분해가 적절해야 한다. 너무 세분화하면 오히려 컨텍스트를 낭비하고, 너무 거칠면 핵심 추론을 건너뛴다. 각 단계가 이전 단계의 결론을 기반으로 다음 단계를 도출하는 논리적 연쇄(Chain)를 형성해야 의미가 있다.

### Tika 앱 적용 예시

**예시 1: 드래그앤드롭 상태 변경 로직**

드래그앤드롭은 Tika 앱에서 가장 복잡한 비즈니스 로직이다. 칼럼 이동에 따라 startedAt, completedAt이 자동으로 설정·초기화되어야 하고, position도 재계산해야 한다. 이런 복합적인 로직을 한 번에 "구현해줘"라고 던지면 조건 누락이 발생하기 쉽다.

```
티켓의 드래그앤드롭 상태 변경 로직을 구현해야 해.
단계별로 생각해줘.

1단계: 어떤 칼럼에서 어떤 칼럼으로 이동했는지 판별
  - source status와 destination status를 비교

2단계: 상태별 자동 필드 업데이트 규칙 정리
  - BACKLOG → TODO: startedAt = now()
  - TODO → BACKLOG: startedAt = null
  - 아무 칼럼 → DONE: completedAt = now()
  - DONE → 아무 칼럼: completedAt = null
  - 위 규칙 외의 이동: startedAt, completedAt 변경 없음

3단계: position 재계산
  - 대상 칼럼에서 드롭 위치의 앞뒤 카드 position 중간값 계산
  - 간격이 1 이하면 해당 칼럼 전체 재정렬 (1024 간격)

4단계: 낙관적 업데이트 적용
  - UI를 먼저 변경하고, API를 비동기 호출

5단계: API 실패 시 롤백
  - 이전 상태로 UI를 되돌리고 에러 토스트 표시

각 단계의 로직을 먼저 설명하고, 그 다음에 코드를 작성해줘.
```

이렇게 요청하면 AI는 각 단계를 분석한 뒤 코드를 작성한다. 특히 2단계에서 규칙을 명시적으로 나열했기 때문에, "DONE에서 IN_PROGRESS로 옮길 때 completedAt 초기화"처럼 놓치기 쉬운 엣지 케이스도 빠뜨리지 않는다.

**예시 2: 오버듀 판정 로직 설계**

```
오버듀 판정 로직을 설계해야 해. 단계별로 분석해보자.

1단계: 오버듀의 정의는?
  → due_date가 오늘 이전이고, status가 DONE이 아닌 티켓

2단계: 타임존 이슈는?
  → 서버는 UTC, 클라이언트는 Asia/Seoul
  → 자정 직전/직후에 판정이 달라질 수 있음

3단계: 판정 위치는 서버? 클라이언트?
  → 서버에서 isOverdue 플래그를 계산해서 내려줄지
  → 클라이언트에서 due_date와 현재 시간을 비교할지

4단계: 날짜 경계 처리는?
  → due_date가 '2026-02-05'이면, 
     2월 5일 23:59:59까지는 오버듀가 아니고
     2월 6일 00:00:00부터 오버듀인가?

각 단계의 결론을 먼저 정리한 뒤, 
그 결론에 맞는 유틸리티 함수를 구현해줘.
```

### 생각의 사슬 vs Plan 모드

Claude Code의 Plan 모드(`Shift+Tab`)도 유사한 효과를 낸다. Plan 모드에서는 Claude가 코드를 작성하기 전에 실행 계획을 먼저 보여주고 개발자의 승인을 기다린다. 생각의 사슬이 프롬프트 수준에서 추론을 유도하는 것이라면, Plan 모드는 도구 수준에서 "계획 → 승인 → 실행"의 흐름을 강제하는 것이다. 복잡한 로직에서는 두 가지를 결합하면 효과적이다. 생각의 사슬로 요청하고, Plan 모드에서 AI의 계획을 리뷰한 뒤, 실행 모드로 전환하여 구현하는 방식이다.

## 4. 역할 부여 프롬프팅 (Role Prompting)

### 개념

역할 부여 프롬프팅은 AI에게 특정 전문가의 역할을 부여하여 해당 관점에서 분석하거나 구현하도록 유도하는 기법이다. "시니어 프론트엔드 개발자로서 리뷰해줘"라고 하면, AI는 그 역할에 맞는 전문 지식, 관심사, 평가 기준을 활성화한다.

역할 부여가 효과적인 이유는 AI의 응답 범위를 좁히기 때문이다. "이 코드 봐줘"보다 "보안 엔지니어로서 이 코드 봐줘"가 더 구체적인 피드백을 이끌어낸다. 전자는 가능한 모든 관점을 얕게 다루지만, 후자는 보안이라는 렌즈에 집중하여 깊은 분석을 제공한다.

### 언제 사용하는가

- 코드 리뷰에서 특정 관점의 피드백이 필요할 때
- 아키텍처 결정에 대한 다각적 분석이 필요할 때
- 성능, 보안, 접근성 등 전문 영역의 검토가 필요할 때
- 주니어 개발자가 시니어의 관점을 학습하고 싶을 때

### 핵심 원칙

역할은 구체적일수록 좋다. "개발자"보다 "5년 차 시니어 프론트엔드 개발자"가, "리뷰해줘"보다 "모바일 환경에서의 UX 관점으로 리뷰해줘"가 더 집중된 응답을 이끌어낸다. 역할을 부여할 때는 해당 역할이 중시하는 가치나 기준을 함께 명시하면 효과가 배가된다.

### Tika 앱 적용 예시

**예시 1: UX 관점의 드래그앤드롭 리뷰**

```
너는 시니어 프론트엔드 개발자야.
Tika 칸반 보드의 드래그앤드롭 UX를 리뷰해줘.
@dnd-kit을 쓰고 있고, 낙관적 업데이트를 적용 중이야.

특히 다음 관점에서 봐줘:
- 드래그 중 시각적 피드백이 충분한가
- 네트워크 지연 시 사용자가 혼란스럽지 않은가
- 모바일 터치 환경에서 오동작 가능성
- 접근성(키보드 네비게이션)은 고려되어 있는가
```

**예시 2: 보안 관점의 API 검토**

```
너는 보안 엔지니어야.
Tika 앱의 API 엔드포인트를 검토해줘.
현재는 인증 없는 단일 사용자 앱이야.

다음을 중점적으로 봐줘:
- 입력값 검증이 충분한가 (SQL Injection, XSS 가능성)
- 에러 응답에서 내부 정보가 노출되지는 않는가
- 추후 멀티 사용자로 확장할 때 미리 대비할 점
```

**예시 3: 테스트 엔지니어 관점**

```
너는 QA 엔지니어야.
Tika 앱의 티켓 상태 변경 플로우에서 
놓칠 수 있는 엣지 케이스를 찾아줘.

현재 구현된 상태 전이:
- BACKLOG ↔ TODO ↔ IN_PROGRESS ↔ DONE
- 어떤 칼럼에서든 어떤 칼럼으로든 이동 가능

특히 startedAt과 completedAt 자동 관리에서 
꼬일 수 있는 시나리오를 중심으로 봐줘.
```

### 역할 부여의 주의점

역할 부여가 과하면 AI가 해당 역할에 과도하게 몰입하여 실용적이지 않은 조언을 할 수 있다. 예를 들어 "보안 전문가"로 역할을 부여하면 MVP 단계에서 불필요한 수준의 보안 조치를 권할 수 있다. 역할 부여 후에도 프로젝트의 맥락(MVP 단계, 단일 사용자 앱 등)을 함께 전달하는 것이 중요하다.

## 5. 제약 조건 프롬프팅 (Constraint Prompting)

### 개념

제약 조건 프롬프팅은 AI가 해야 할 것뿐 아니라, 하지 말아야 할 것을 명시적으로 지정하는 기법이다. AI 코딩 도구의 가장 흔한 문제 중 하나는 "요청하지 않은 것까지 해버리는 것"이다. 파일 하나 수정을 요청했는데 관련 파일 세 개를 함께 바꾸거나, 기존 코드를 "개선"한다며 동작하던 로직을 변경해버린다.

제약 조건은 이 문제에 대한 직접적인 해법이다. 켄트 벡이 증강코딩에서 강조한 "요청한 것만 구현, 합리적으로 보이는 추가 기능 금지" 원칙을 프롬프트 수준에서 실현하는 방법이기도 하다.

### 언제 사용하는가

- AI가 범위를 넘어서는 경향이 있을 때
- 기존 코드를 건드리지 않고 특정 부분만 수정해야 할 때
- 특정 라이브러리나 기술을 사용하거나 배제해야 할 때
- 성능, 번들 크기 등 구체적인 기술적 제약이 있을 때

### 핵심 원칙

제약 조건은 긍정형("~을 사용해")과 부정형("~하지 마")을 모두 활용한다. 특히 부정형 제약이 중요하다. AI는 기본적으로 "최선의 결과를 내려는" 성향이 있어서, 명시적으로 금지하지 않으면 "좋아 보이는" 변경을 스스로 추가한다. 제약 조건은 `[제약 조건]` 같은 명확한 레이블로 구분하면 AI가 더 잘 인식한다.

### Tika 앱 적용 예시

**예시 1: 범위 제한이 있는 API 수정**

```
PATCH /api/tickets/:id 엔드포인트를 수정해줘.

[수정 내용]
- Zod 스키마에 dueDate 검증 추가 (오늘 이후 날짜만 허용)

[제약 조건]
- title, description, priority, plannedStartDate, dueDate만 수정 가능
- status, startedAt, completedAt은 이 엔드포인트로 수정 불가
- 존재하지 않는 필드가 body에 오면 무시 (에러 아님)
- route.ts 파일만 수정. 다른 파일은 건드리지 마
- 기존 테스트 파일은 수정하지 마
```

`[제약 조건]` 블록이 없으면 AI가 "테스트도 업데이트해야겠다"며 테스트 파일까지 수정하거나, "status도 같이 변경할 수 있으면 좋겠다"며 요청 범위를 넓히는 일이 빈번하다.

**예시 2: 기술 스택 제약**

```
TicketModal에 삭제 확인 다이얼로그를 추가해줘.

[제약 조건]
- 새로운 라이브러리를 설치하지 마
- 기존 ConfirmDialog 컴포넌트를 재사용할 것
- 삭제 API 호출은 이미 useTickets.remove에 구현되어 있음. 새로 만들지 마
- 스타일링은 Tailwind CSS만 사용
- 인라인 스타일이나 CSS 모듈을 사용하지 마
```

**예시 3: 성능 제약**

```
보드 조회 API (GET /api/tickets)를 최적화해줘.

[제약 조건]
- 현재 쿼리 구조(단일 SELECT)를 유지할 것
- 캐싱 레이어를 추가하지 마 (MVP 단계)
- 인덱스 추가는 허용하되, idx_ 접두사로 네이밍
- 응답 형태(JSON 구조)를 변경하지 마. 프론트엔드에 영향이 감
```

### 제약 조건과 CLAUDE.md

퓨샷과 마찬가지로, 프로젝트 전반에 적용되는 제약 조건은 CLAUDE.md에 정리해둘 수 있다.

```markdown
# CLAUDE.md

## 변경 범위 규칙
- 요청한 파일만 수정할 것
- 테스트 파일은 명시적으로 요청하지 않으면 수정하지 않음
- 기존 인터페이스(API 응답 구조, Props 타입)를 변경하지 않음

## 기술 스택 규칙
- 새로운 npm 패키지 설치 금지 (사전 승인 필요)
- Tailwind CSS만 사용, CSS-in-JS 금지
- any 타입 사용 금지
```

## 6. 단계적 분해 프롬프팅 (Step-by-Step Decomposition)

### 개념

단계적 분해는 하나의 큰 작업을 여러 개의 작은 작업으로 나누어 순차적으로 요청하는 기법이다. 3.2절에서 다룬 "작은 단위로 쪼개기"를 프롬프트 수준에서 실현하는 방법이다.

생각의 사슬이 하나의 프롬프트 안에서 단계를 밟는 것이라면, 단계적 분해는 여러 번의 프롬프트에 걸쳐 단계를 밟는 것이다. 각 단계를 완료하고 검증한 뒤 다음 단계로 넘어가기 때문에, 중간에 문제가 생겨도 되돌릴 범위가 작다.

### 언제 사용하는가

- 하나의 기능이 여러 레이어(DB → API → UI)를 관통할 때
- 각 단계의 결과를 검증해야 안전하게 다음으로 넘어갈 수 있을 때
- TDD 사이클(Red → Green → Refactor)을 따를 때
- 한 번에 요청하면 AI가 방향을 잘못 잡을 위험이 클 때

### 핵심 원칙

각 단계는 독립적으로 검증 가능해야 한다. "Zod 스키마를 정의해줘"가 하나의 단계라면, 해당 스키마에 유효/무효 입력을 넣어보는 것이 검증이다. 검증 없이 다음 단계로 넘어가면 단계적 분해의 의미가 없다. 또한 각 단계의 크기가 균일해야 한다. 1단계가 5줄짜리 스키마 정의인데 2단계가 200줄짜리 API 전체 구현이면, 분해가 제대로 된 것이 아니다.

### Tika 앱 적용 예시

**예시: 티켓 생성 기능을 6단계로 분해**

```
티켓 생성 기능을 단계별로 구현하자.
한 단계씩 진행할게. 지금은 1단계만 해줘.

[1단계] Zod 스키마 정의 — CreateTicketInput
→ 검증: 유효/무효 입력에 대해 parse 테스트
```

1단계 완료 및 검증 후:

```
1단계 확인 완료. 2단계 진행해줘.

[2단계] POST /api/tickets 라우트 핸들러 작성
→ 1단계에서 만든 Zod 스키마로 요청 body를 검증
→ 검증 통과 시 DB에 삽입, 실패 시 400 반환
→ 검증: Thunder Client로 201/400 응답 확인
```

2단계 완료 및 검증 후:

```
2단계 확인 완료. 3단계 진행.

[3단계] position 자동 계산 로직
→ 같은 status 내에서 가장 작은 position - 1024 할당
→ 티켓이 없으면 기본값 0
→ 검증: 기존 Backlog 티켓이 있을 때 새 티켓의 position이 더 작은지 확인
```

```
3단계 확인 완료. 4단계 진행.

[4단계] 에러 핸들링 보강
→ DB 연결 실패 시 500 응답
→ 예상치 못한 에러에 대한 try-catch
→ 검증: 의도적으로 DB 연결을 끊고 500 응답 확인
```

```
4단계 확인 완료. 5단계 진행.

[5단계] 통합 테스트 작성
→ 정상 생성, 필수 필드 누락, 유효하지 않은 값 등 케이스 포함
→ 검증: jest --verbose로 모든 테스트 통과 확인
```

```
5단계 확인 완료. 6단계 진행.

[6단계] 프론트엔드 연결
→ useTickets 훅에 create 함수 추가
→ TicketForm에서 onSubmit 시 create 호출
→ 성공 시 Backlog 칼럼에 새 카드 표시
→ 검증: 브라우저에서 전체 플로우 테스트
```

### 단계적 분해와 Git 커밋

각 단계가 끝나면 커밋하는 습관을 들이면 효과가 배가된다. 3단계에서 문제가 발생하면 2단계 커밋으로 되돌릴 수 있다. Claude Code의 `/rewind` 명령어와 결합하면 더욱 안전하다.

```
# 단계별 커밋 예시
git commit -m "feat(ticket): CreateTicketInput Zod 스키마 정의"
git commit -m "feat(ticket): POST /api/tickets 라우트 핸들러"
git commit -m "feat(ticket): position 자동 계산 로직"
git commit -m "feat(ticket): 에러 핸들링 보강"
git commit -m "test(ticket): 티켓 생성 통합 테스트"
git commit -m "feat(ticket): 프론트엔드 useTickets.create 연결"
```

## 기법 선택 가이드

어떤 기법을 쓸지 고민된다면, 다음 기준으로 판단한다.

| 상황 | 추천 기법 | 이유 |
|------|----------|------|
| 단순 CRUD, 한 줄짜리 수정 | 제로샷 | 예시나 추론이 불필요할 만큼 명확 |
| 코드 스타일·패턴 통일 | 퓨샷 | 예시가 말보다 빠르고 정확 |
| 복잡한 비즈니스 규칙 | 생각의 사슬 | 중간 추론을 거쳐야 조건 누락 방지 |
| 특정 관점의 리뷰·분석 | 역할 부여 | 관점을 좁혀야 깊이가 생김 |
| AI의 과잉 행동 방지 | 제약 조건 | 명시적 금지 없이는 범위가 넓어짐 |
| 대규모 기능 구현 | 단계적 분해 | 검증 가능한 단위로 나눠야 안전 |

## 기법의 조합

실무에서는 하나의 기법만 단독으로 사용하는 경우가 드물다. 복잡한 작업일수록 여러 기법을 겹쳐서 사용하면 효과가 배가된다.

**조합 예시: 역할 부여 + 생각의 사슬 + 제약 조건**

```
너는 시니어 백엔드 개발자야. (역할 부여)

Done 칼럼의 24시간 자동 숨김 로직을 구현해야 해.
단계별로 분석해줘. (생각의 사슬)

1단계: 숨김 기준 시간을 서버에서 계산할지, 클라이언트에서 계산할지 결정
2단계: DB 쿼리에 시간 필터를 넣을지, 애플리케이션 레벨에서 필터링할지 결정
3단계: 결정된 방식으로 구현

[제약 조건] (제약 조건)
- 기존 GET /api/tickets의 응답 구조를 변경하지 마
- 새로운 엔드포인트를 추가하지 마
- cron job이나 스케줄러를 사용하지 마 (서버리스 환경)
```

이 프롬프트는 세 가지 기법이 자연스럽게 결합되어 있다. 역할 부여로 백엔드 관점에 집중시키고, 생각의 사슬로 설계 결정을 단계적으로 도출하며, 제약 조건으로 서버리스 환경의 현실적 한계를 반영한다.

**조합 예시: 퓨샷 + 제약 조건 + 단계적 분해**

```
다음 패턴으로 API 엔드포인트별 통합 테스트를 작성해줘. (퓨샷)

[패턴 예시]
describe('POST /api/tickets', () => {
  it('should return 201 with valid input', async () => {
    const res = await request(app)
      .post('/api/tickets')
      .send({ title: '새 티켓' });
    expect(res.status).toBe(201);
    expect(res.body.data).toHaveProperty('id');
  });
});

[제약 조건] (제약 조건)
- supertest 라이브러리만 사용
- 각 테스트는 독립적 (DB 상태에 의존하지 않음)
- 테스트 데이터는 한국어로 작성

지금은 POST /api/tickets에 대한 테스트만 작성해줘. (단계적 분해)
나머지 엔드포인트는 다음 단계에서 진행할게.
```

## 실전 팁: 프롬프트도 반복이다

좋은 프롬프트는 한 번에 완성되지 않는다. 코드를 리팩토링하듯, 프롬프트도 결과를 보고 개선해야 한다.

첫 번째 시도에서 원하는 결과가 나오지 않았다면, 다음을 점검한다.

1. **지시가 모호하지 않았는가?** — 제로샷을 더 구체적으로 다듬는다
2. **패턴을 말로 설명하고 있었는가?** — 퓨샷으로 예시를 보여준다
3. **조건이 복잡한데 한 번에 요청했는가?** — 생각의 사슬로 단계를 나눈다
4. **AI가 엉뚱한 방향으로 갔는가?** — 제약 조건을 추가한다
5. **작업이 너무 크지 않았는가?** — 단계적 분해로 쪼갠다

프롬프트 엔지니어링은 AI를 조종하는 기술이 아니다. AI와의 의사소통을 정교하게 다듬는 과정이다. 그리고 이 과정에서 개발자가 얻는 것은 AI의 더 나은 응답만이 아니다. "내가 무엇을 원하는가"를 스스로 명확히 정리하는 능력, 즉 설계 사고력 자체가 함께 성장한다.
