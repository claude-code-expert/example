# 하네스 엔지니어링: 에이전트를 위한 품질 보증 구조 설계

## 프롬프트, 컨텍스트, 그리고 하네스

앞 절에서 프롬프트 엔지니어링("어떻게 물어볼 것인가")과 컨텍스트 엔지니어링("AI가 무엇을 보게 할 것인가")을 살펴보았다. 이 두 기법은 단일 세션, 단일 작업 범위에서 AI의 결과 품질을 끌어올리는 데 효과적이다.

그런데 실제 프로젝트는 단일 세션으로 끝나지 않는다. Tika 칸반 보드를 처음부터 완성하려면 데이터 모델 설계, 7개의 API 엔드포인트 구현, 드래그앤드롭 프론트엔드, 테스트, 배포까지 수십 개의 작업이 필요하다. 이 작업들을 AI 에이전트에게 위임하면 어떤 일이 벌어질까?

Anthropic 엔지니어링 팀(Labs)은 이 질문에 직접 부딪히며 두 가지 근본적인 실패 모드를 발견했다.

---

## 에이전트가 혼자 달리면 부딪히는 두 가지 벽

### 실패 모드 1: 컨텍스트 불안 (Context Anxiety)

앞 절에서 컨텍스트 부패(Context Rot)를 다루었다. 컨텍스트 윈도우에 토큰이 쌓일수록 모델의 정보 회수 능력이 떨어지는 현상이다. 하네스 엔지니어링에서 다루는 컨텍스트 불안은 이보다 한 단계 더 심각한 문제다.

일부 모델은 컨텍스트 윈도우가 한계에 가까워지면 작업을 조기에 마무리하려는 경향을 보인다. 마치 마감 시간이 다가오면 급하게 일을 끝내려는 사람처럼, AI가 아직 완성되지 않은 기능을 "완료"로 선언해 버린다.

Tika 프로젝트에서 이런 상황을 상상해 보자. 드래그앤드롭 구현을 요청했는데, AI가 칼럼 간 이동까지만 구현하고 "완료"를 선언한다. startedAt/completedAt 자동 관리 로직은 `// TODO: 구현 필요`라는 주석으로 남아 있다. position 재계산 로직은 아예 빠져 있다. 하지만 AI는 "드래그앤드롭 기능을 성공적으로 구현했습니다"라고 자신 있게 보고한다.

Anthropic 팀은 초기 실험에서 Sonnet 4.5 모델이 이 경향을 강하게 보였다고 밝혔다. 컨텍스트 압축(`/compact`)으로도 충분하지 않아서, 컨텍스트 리셋 — 컨텍스트 윈도우를 완전히 비우고 새로운 에이전트를 시작하되, 이전 에이전트의 상태를 구조화된 핸드오프 문서로 전달하는 방식 — 이 필수적이었다고 한다. 다만 이후 등장한 Opus 모델은 이 경향이 크게 줄어들어, 하네스 설계에서 컨텍스트 리셋을 제거할 수 있었다.

### 실패 모드 2: 자기 평가 불능 (Self-Evaluation Blindspot)

더 근본적인 문제가 있다. 에이전트에게 자기 작업을 평가하라고 하면, 자신의 결과물을 과하게 칭찬한다.

"이 API가 제대로 동작합니까?" → "네, 잘 동작합니다."

실제로 열어보면 에러 케이스 처리가 없거나, 항상 200을 반환하거나, 핵심 로직이 stub 상태로 남아 있다. 하지만 생성자가 자기 코드를 검토하면, 문제를 발견하고도 "별로 중요하지 않다"고 결론짓는 현상이 반복된다.

이 문제는 UI나 디자인처럼 주관적인 영역에서 더 심각하다. "이 레이아웃이 보기 좋은가?"라는 질문에 대해 객관적 기준이 없으면, AI는 기본적으로 긍정적인 평가를 내린다. 자신이 만든 코드에는 더욱 그렇다.

Anthropic 팀의 표현을 빌리면, "생성자 에이전트를 자기 작업에 비판적으로 만드는 것보다, 독립된 평가자 에이전트를 회의적으로 튜닝하는 것이 훨씬 다루기 쉽다."

## 해결책: GAN에서 영감을 받은 멀티 에이전트 구조

Anthropic Labs 팀은 이 두 문제를 동시에 해결하기 위해 GAN(Generative Adversarial Networks)의 구조에서 영감을 얻었다. GAN에서 생성자(Generator)와 판별자(Discriminator)가 서로 경쟁하며 품질을 높이듯, AI 에이전트도 "만드는 역할"과 "평가하는 역할"을 분리하면 품질이 크게 향상된다.

이것이 하네스(Harness)다. 에이전트가 자유롭게 달리되, 올바른 방향으로 달리도록 제어하는 구조적 장치다. 
하네스는 본래 말에게 채우는 끌채 장비를 뜻한다. 말이 자유롭게 달리되, 그 힘이 마차를 끌어당기는 올바른 방향으로 전달되게 하는 장치다. AI 에이전트도 마찬가지다. 자율적으로 코드를 생성하되, 품질이 보장되는 방향으로 달리게 하는 구조적 장치가 필요하다. AI 에이전트가 장시간 자율적으로 작업할 때, 올바른 방향으로 작동하도록 제어하는 구조적 장치를 말한다. 

```
사용자 프롬프트 (1~4문장)
       │
       ▼
┌──────────────┐
│   Planner    │  짧은 프롬프트를 상세 스펙으로 확장
│              │  스프린트 단위로 분해
└──────┬───────┘
       │
       ▼
┌──────────────┐        ┌──────────────┐
│  Generator   │◄──────►│  Evaluator   │
│              │        │              │
│ 스프린트 계약 │  피드백 │ 스프린트 계약 │
│ 협상 후 구현  │◄───────│ 검증 후 판정  │
│              │        │              │
│ 결과: 코드   │        │ 결과: QA 보고 │
└──────┬───────┘        └──────────────┘
       │
       │  FAIL → 피드백 반영 후 재시도
       │  PASS → 다음 스프린트
       ▼
┌──────────────┐
│  완성된 앱    │
└──────────────┘
```

세 에이전트 각각의 역할을 Tika 프로젝트에 대입하면 다음과 같다.

**Planner** — "Tika 칸반 보드를 만들어줘"라는 한 줄의 요청을 받아, PRD 수준의 상세 스펙을 작성한다. 기능을 스프린트 단위로 분해하고, 각 스프린트의 범위를 정의한다. 기술적 구현 세부사항은 최소화하고, "무엇을 만들 것인가"에 집중한다. 잘못된 기술 세부사항이 스펙에 들어가면 하위 에이전트에 연쇄적으로 영향을 미치기 때문이다.

**Generator** — 스프린트 단위로 코드를 구현한다. 각 스프린트에서 하나의 기능을 완성하고, 자체 검증 후 Evaluator에게 넘긴다.

**Evaluator** — Generator가 "완료했다"고 보고한 코드를 실제로 실행하면서 검증한다. Anthropic 팀은 Evaluator에게 Playwright MCP를 제공하여 실행 중인 앱을 직접 클릭하고 탐색하게 했다. 각 스프린트의 평가 기준에 따라 PASS/FAIL을 판정하고, FAIL 시 구체적인 수정 지시를 Generator에게 전달한다.

## STEP 1: 평가 기준 정의 — "좋은 결과"를 측정 가능하게

하네스 설계에서 가장 먼저 해야 할 일은 "좋은 결과"를 구체적으로 정의하는 것이다. 주관적인 판단을 그대로 두면 에이전트도 개발자도 결과를 일관되게 평가할 수 없다.

Anthropic 팀은 프론트엔드 실험에서 4가지 평가 기준을 사용했다. 흥미로운 점은 기준의 가중치 설정이다.

| 기준 | 핵심 질문 | 가중치 |
|------|-----------|--------|
| 디자인 품질 | 색상, 타이포, 레이아웃이 하나의 일관된 분위기를 만드는가? | ★★★★★ |
| 독창성 | 라이브러리 기본값이나 AI 특유의 패턴을 그대로 쓴 것은 아닌가? | ★★★★★ |
| 크래프트 | 타이포그래피 계층, 간격 일관성, 색상 대비가 기본 이상인가? | ★★★ |
| 기능성 | 사용자가 UI를 이해하고 주요 동작을 완료할 수 있는가? | ★★★ |

왜 디자인 품질과 독창성의 가중치가 더 높은가? AI는 크래프트와 기능성은 기본적으로 잘 해낸다. 문제는 항상 "AI 티가 나는" 평범한 결과물이다. 보라색 그라데이션 위에 흰색 카드, 모든 SaaS 랜딩 페이지에서 본 듯한 레이아웃. 평가 기준의 가중치 자체가 에이전트의 방향을 조정한다.

### Tika 프로젝트 적용 예시

Tika는 프론트엔드 디자인 프로젝트가 아니라 풀스택 앱이므로, 평가 기준을 백엔드와 전체 품질에 맞게 조정해야 한다.

```markdown
## Tika 프로젝트 품질 평가 기준

### 1. 기능 완성도 (★★★★★)
- 스펙에 정의된 기능이 end-to-end로 동작하는가?
- stub 함수나 TODO 주석이 남아있지 않은가?
- 에러 케이스(null, 빈 배열, 네트워크 오류)를 처리하는가?
- "동작하는 척"이 아니라 실제 사용 시나리오에서 작동하는가?

### 2. 데이터 무결성 (★★★★★)
- 드래그앤드롭 시 startedAt/completedAt 자동 관리 규칙이 정확한가?
- position 재계산이 올바른가? (간격이 1 이하일 때 재정렬 포함)
- Done 칼럼의 24시간 필터가 정확히 동작하는가?
- 낙관적 업데이트 실패 시 롤백이 작동하는가?

### 3. 타입 안전성 (★★★)
- any 타입 사용이 없는가?
- TypeScript 컴파일 에러가 없는가? (npx tsc --noEmit)
- Zod 스키마와 Drizzle 스키마가 일치하는가?

### 4. 테스트 커버리지 (★★★)
- 핵심 비즈니스 로직에 단위 테스트가 있는가?
- 엣지 케이스가 테스트되었는가?
- npm test가 0 failures로 통과하는가?
```

가중치가 높은 항목에 주목하라. 기능 완성도와 데이터 무결성이 최우선이다. Tika에서 가장 복잡한 비즈니스 규칙은 상태 전이에 따른 타임스탬프 자동 관리인데, 이 부분이 깨지면 앱 전체의 신뢰성이 무너진다. 타입 안전성과 테스트 커버리지는 AI가 기본적으로 잘 해내는 영역이므로 가중치를 낮게 둔다.

## STEP 2: 스프린트 계약 — 코딩 전에 "완료"를 합의하기

Anthropic 팀의 하네스에서 가장 인상적인 설계 중 하나가 스프린트 계약(Sprint Contract)이다. Generator가 코딩을 시작하기 전에, Generator와 Evaluator가 "이번 스프린트에서 무엇을 만들고, 어떻게 검증할 것인가"를 먼저 합의한다.

왜 이것이 필요한가? 제품 스펙은 의도적으로 고수준으로 작성된다(Planner가 기술 세부사항을 과도하게 명시하면 하위 에이전트에 오류가 전파되므로). 고수준 스펙과 실제 구현 사이의 간극을 메우는 것이 스프린트 계약의 역할이다.

### Tika 스프린트 계약 예시

```markdown
# Sprint 3 계약: 드래그앤드롭 구현

## Generator 제안
이번 스프린트에서 @dnd-kit을 사용하여 칼럼 간, 칼럼 내 
카드 드래그앤드롭을 구현합니다.

### 구현 범위
- [ ] 칼럼 간 카드 이동 (상태 변경)
- [ ] 같은 칼럼 내 카드 순서 변경
- [ ] 낙관적 업데이트 (UI 즉시 반영, API 비동기 호출)
- [ ] API 실패 시 롤백

### 완료 조건
1. Backlog에서 TODO로 카드를 드래그하면 status가 변경된다
2. TODO 내에서 카드 순서를 변경하면 position이 재계산된다
3. 네트워크 에러 시 카드가 원래 위치로 돌아간다
4. npm test가 통과한다

---

## Evaluator 검토
Generator의 제안에 다음 조건을 추가합니다:

### 추가 완료 조건
5. BACKLOG → TODO 이동 시 startedAt이 설정되는지 확인
6. TODO → BACKLOG 복귀 시 startedAt이 null로 초기화되는지 확인
7. 아무 칼럼 → DONE 이동 시 completedAt이 설정되는지 확인
8. DONE → 아무 칼럼 이동 시 completedAt이 null로 초기화되는지 확인
9. position 간격이 1 이하로 좁아졌을 때 칼럼 전체 재정렬이 
   동작하는지 확인

### 테스트 방법
- Playwright로 실행 중인 앱에서 직접 드래그앤드롭 수행
- DB 상태를 API 호출로 검증 (GET /api/tickets 후 필드 값 확인)
- 네트워크 차단 후 롤백 동작 확인

위 9개 조건 중 하나라도 FAIL이면 스프린트 전체를 FAIL 처리합니다.

---

## 합의
Generator: 동의합니다. 구현을 시작합니다.
```

실제 Anthropic 팀의 실험에서, 한 스프린트의 계약에 27개의 검증 기준이 포함된 사례도 있었다. 이 정도로 구체적이어야 Evaluator가 "대충 봐주고 PASS" 하는 것을 방지할 수 있다.

## STEP 3: Evaluator 에이전트 — 까다로운 QA 엔지니어 만들기

Evaluator는 하네스의 핵심이다. Generator가 만든 결과물을 독립적으로 검증하는 역할이다. Anthropic 팀은 Evaluator에게 Playwright MCP를 제공하여 실행 중인 앱을 직접 조작하고 스크린샷을 찍으며 검증하게 했다.

### Evaluator 프롬프트 설계

```markdown
# evaluator.md

당신은 까다로운 시니어 엔지니어이자 QA 전문가입니다.
Generator 에이전트가 제출한 결과물을 검증합니다.

## 평가 원칙
- Generator가 "완료했다"고 주장해도 직접 실행해서 확인하라
- 표면적 동작이 아니라 실제 사용 시나리오를 기준으로 판단하라
- 의심스러운 부분은 코드를 열어서 stub인지 확인하라
- 버그를 발견하고도 "별로 중요하지 않다"고 결론짓지 마라
- 관대하게 평가하는 것은 당신의 역할이 아니다

## 검증 방법
1. 스프린트 계약의 완료 조건을 하나씩 검증
2. 각 조건에 대해 PASS/FAIL과 구체적 근거 제시
3. FAIL 시 정확한 파일 경로와 라인 번호 포함
4. 수정 방향을 구체적으로 제시

## 결과 보고 형식

### QA 보고서 — Sprint N

**종합 판정**: PASS / FAIL

| 조건 | 결과 | 근거 |
|------|------|------|
| 1. 칼럼 간 이동 시 status 변경 | PASS | Backlog→TODO 이동 확인, DB 값 검증 |
| 5. TODO 이동 시 startedAt 설정 | FAIL | startedAt이 null인 채로 남아있음 |

**Generator에게 전달할 수정 지시:**
1. `app/api/tickets/reorder/route.ts:47` — TODO로 이동할 때 
   startedAt 설정 로직이 누락됨. 
   status가 'TODO'이고 이전 status가 'BACKLOG'일 때 
   startedAt = new Date()를 추가할 것.
```

### Evaluator가 실제로 발견하는 문제의 유형

Anthropic 팀의 실험에서 Evaluator가 발견한 버그 사례들을 보면, 단순한 문법 오류가 아니라 "직접 써봐야 알 수 있는" 수준의 문제들이다.

| 스프린트 계약 기준 | Evaluator 발견 사항 |
|-------------------|-------------------|
| 사각형 채우기 도구가 클릭-드래그로 영역을 채울 수 있어야 함 | FAIL — 도구가 드래그 시작/끝 지점에만 타일을 배치하고, 영역을 채우지 않음. `fillRectangle` 함수가 존재하지만 mouseUp 이벤트에서 호출되지 않음 |
| 사용자가 배치된 엔티티 스폰 포인트를 선택하고 삭제할 수 있어야 함 | FAIL — `LevelEditor.tsx:892`의 Delete 키 핸들러가 `selection`과 `selectedEntityId`가 모두 설정되어야 동작함. 하지만 엔티티 클릭 시 `selectedEntityId`만 설정됨 |
| 사용자가 API로 애니메이션 프레임 순서를 변경할 수 있어야 함 | FAIL — `PUT /frames/reorder` 라우트가 `/{frame_id}` 라우트 뒤에 정의됨. FastAPI가 'reorder'를 frame_id 정수로 파싱하려다 422 에러 반환 |

이런 버그들은 코드를 읽는 것만으로는 발견하기 어렵다. 실제로 앱을 실행하고, 사용자처럼 클릭하고, 결과를 확인해야 드러난다. 이것이 Evaluator에게 Playwright MCP를 제공하는 이유다.

## STEP 4: Evaluator 튜닝 — 처음부터 완벽한 Evaluator는 없다

Anthropic 팀이 명시적으로 인정한 사실이 있다: "처음에 Claude는 형편없는 QA 에이전트였다." 초기 실행에서 Evaluator는 진짜 문제를 발견하고도 "별로 중요하지 않다"고 결론짓거나, 표면적인 검증만 하고 깊이 있는 엣지 케이스는 건너뛰는 경향을 보였다.

이를 해결하는 과정이 튜닝 루프(Tuning Loop)다. 하네스 설계에서 가장 시간이 많이 걸리지만, 가장 중요한 작업이다.

### 튜닝 프로세스

```
1. 하네스 실행
       │
       ▼
2. Evaluator 로그 읽기
       │
       ▼
3. "내 판단과 다른 사례" 찾기
   - Evaluator가 PASS했는데 실제로는 broken
   - Evaluator가 사소한 것에 FAIL해서 재작업 낭비
       │
       ▼
4. 해당 사례를 Evaluator 프롬프트에 퓨샷(Few-Shot)으로 추가
       │
       ▼
5. 다시 실행 → 1번으로
```

### 퓨샷 사례 추가 예시

```markdown
## 판단 사례 (Evaluator 프롬프트에 추가)

### FAIL이 맞는 사례
상황: Generator가 "티켓 생성 API 완료"라고 보고.
      코드를 보니 position 계산 로직에 
      `// TODO: 기존 티켓 position 고려` 주석이 있음.
판정: FAIL — stub 코드가 포함된 기능은 완료가 아님.

### PASS가 맞는 사례
상황: 오버듀 뱃지 색상이 디자인 스펙보다 약간 진함. (#FF0000 vs #EF4444)
판정: PASS — 핵심 기능에 영향 없는 미세한 스타일 차이는 통과.

### 잘못된 PASS 사례 (이렇게 판단하지 말 것)
상황: GET /api/tickets가 Done 칼럼의 24시간 필터를 적용하지 않고 
      모든 Done 티켓을 반환함.
잘못된 판정: "전반적으로 잘 동작하므로 PASS"
올바른 판정: FAIL — 스펙에 명시된 비즈니스 규칙 위반.
             24시간 필터는 Done 칼럼의 핵심 동작임.
```

Anthropic 팀은 "몇 라운드의 튜닝 루프를 거쳐야 Evaluator의 판단이 합리적인 수준에 도달했다"고 밝혔다. 즉, 하네스를 한 번 만들어서 바로 쓰는 것이 아니라, 반복적으로 개선해야 한다.

## STEP 5: 전체 하네스 실행 — 비용 대 품질

Anthropic 팀의 실험 결과는 명확한 트레이드오프를 보여준다. "2D 레트로 게임 메이커"를 동일한 한 줄 프롬프트로 생성한 결과:

| 접근 방식 | 소요 시간 | 비용 | 결과 |
|-----------|---------|------|------|
| 단일 에이전트 (Solo) | 20분 | $9 | 겉보기엔 동작하나, 핵심 기능 broken |
| 3-에이전트 하네스 | 6시간 | $200 | 실제 플레이 가능한 앱 완성 |

20배 이상의 비용 차이지만, 결과의 차이는 "겉보기에 동작하는 것"과 "실제로 동작하는 것" 사이의 차이다. 단일 에이전트는 레벨 에디터와 스프라이트 에디터를 만들었지만, 실제 게임 플레이에서 엔티티가 입력에 반응하지 않았다. 3-에이전트 하네스는 같은 프롬프트에서 출발하여 16개 기능, 10개 스프린트의 스펙을 자동 생성하고, AI 기능 통합(Claude를 이용한 스프라이트 자동 생성)까지 포함한 앱을 완성했다.

### 언제 하네스를 쓰고, 언제 쓰지 않는가

모든 작업에 3-에이전트 하네스가 필요하지는 않다. Anthropic 팀이 강조한 원칙이 있다:

> "하네스의 모든 컴포넌트는 '모델이 혼자 못 하는 것'에 대한 가정을 인코딩한다. 그 가정은 항상 검증해야 한다."

모델의 능력이 올라가면 불필요해지는 컴포넌트가 생긴다. 실제로 Opus 모델이 등장하면서 컨텍스트 리셋이 불필요해졌고, 하네스 비용이 $200에서 $124로 줄었다.

| 작업 유형 | 권장 접근 |
|-----------|---------|
| 단순 CRUD API 하나 | 단일 에이전트 + 테스트 스크립트로 충분 |
| 복잡한 비즈니스 로직 (Tika의 상태 전이) | Generator + Evaluator (2-에이전트) |
| 멀티 기능 풀스택 앱 (처음부터 완성) | Planner + Generator + Evaluator (3-에이전트) |
| 디자인이 중요한 프론트엔드 | Generator + Evaluator (디자인 기준 포함) |

## STEP 6: 간소화 판단 — 한 번에 하나씩

Anthropic 팀이 직접 인정한 실수가 있다. 하네스 구조를 한꺼번에 급진적으로 줄였다가, 성능이 복원되지 않았다는 것이다.

올바른 간소화 방법은 컴포넌트를 한 번에 하나씩 제거하고, 결과를 비교하는 것이다.

```
✅ 올바른 방법:
   Planner 제거 → 결과 비교 → 품질 유지됨 → Planner 불필요
   Evaluator 제거 → 결과 비교 → 품질 하락 → Evaluator 필수
   
❌ 잘못된 방법:
   Planner + Evaluator + Sprint 구조 모두 제거 
   → 결과 하락 → 어디서 망가졌는지 알 수 없음
```

각 컴포넌트의 간소화 판단 기준을 정리하면 다음과 같다.

| 컴포넌트 | 제거 가능한 경우 | 유지해야 하는 경우 |
|---------|----------------|------------------|
| Planner | 사용자가 이미 상세 스펙을 제공할 때 | 짧은 프롬프트에서 스코프 정의가 필요할 때 |
| Sprint 구조 | 모델이 긴 작업을 일관되게 완주할 때 | 컨텍스트 불안이 발생할 때 |
| Context Reset | 컨텍스트 압축으로 충분할 때 | 압축 후에도 조기 완료 선언이 반복될 때 |
| Evaluator | 작업이 단순하고 모델 능력 범위 안일 때 | 복잡하거나 주관적 품질이 중요할 때 |

## 하네스 엔지니어링과 책의 다른 개념들

하네스 엔지니어링은 이 책에서 다룬 여러 개념을 하나로 엮는 상위 구조다.

**프롬프트 엔지니어링과의 관계** — Evaluator 프롬프트를 튜닝하는 과정은 퓨샷 프롬프팅과 제약 조건 프롬프팅의 직접적인 적용이다. "관대하게 평가하지 마"라는 제약, 올바른/잘못된 판정 사례를 보여주는 퓨샷, "시니어 QA 엔지니어로서"라는 역할 부여가 모두 동원된다.

**컨텍스트 엔지니어링과의 관계** — 스프린트 계약과 핸드오프 문서는 컨텍스트 엔지니어링의 "구조화된 메모(Structured Note-Taking)" 기법이다. 에이전트 간의 소통은 파일을 통해 이루어지며, 이는 컨텍스트 윈도우 외부에 상태를 유지하는 메모리 전략이다.

**TDD와의 관계** — 스프린트 계약의 "완료 조건"은 사실상 테스트 케이스의 자연어 버전이다. "BACKLOG → TODO 이동 시 startedAt이 설정되는지 확인"은 `expect(ticket.startedAt).not.toBeNull()`을 사람 말로 표현한 것이다. Evaluator가 이 조건을 실행해서 PASS/FAIL을 판정하는 것은, CI에서 테스트가 통과/실패하는 것과 같은 구조다.

**증강코딩과의 관계** — 켄트 벡이 강조한 "AI가 앞서 나가지 못하게 하는 것"은 하네스의 Sprint 구조와 직결된다. 한 번에 하나의 기능만 구현하고, 검증 후 다음으로 넘어가는 흐름은 Small, Safe Steps의 자동화된 버전이다.

## 핵심 원칙 요약

1. **평가 기준을 먼저 정의하라** — "좋은 결과"를 주관적으로 두지 않는다. 에이전트가 채점할 수 있는 구체적 기준을 먼저 작성한다.
2. **만드는 역할과 평가하는 역할을 분리하라** — 자기 작업을 자기가 평가하게 하면 항상 관대해진다. 독립된 Evaluator가 검증해야 한다.
3. **코딩 전에 "완료"를 합의하라** — Sprint Contract로 Generator와 Evaluator가 검증 기준을 먼저 협상한다.
4. **Evaluator를 반복 튜닝하라** — 처음부터 완벽한 Evaluator는 없다. 로그를 읽고, 판단이 틀린 사례를 찾아, 퓨샷으로 추가하는 루프를 돌려야 한다.
5. **간소화는 한 번에 하나씩** — 컴포넌트를 한꺼번에 제거하면 어디서 망가졌는지 알 수 없다.
6. **모델이 바뀌면 하네스를 재검토하라** — 모델 능력이 올라가면 불필요한 컴포넌트가 생긴다. 하네스는 모델의 한계에 대한 가정을 인코딩한 것이므로, 그 가정을 주기적으로 검증해야 한다.
