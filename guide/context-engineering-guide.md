# 컨텍스트 엔지니어링: 프롬프트를 넘어서

## 프롬프트 엔지니어링의 한계

앞 절에서 제로샷, 퓨샷, 생각의 사슬 같은 프롬프팅 기법을 살펴보았다. 이 기법들은 AI에게 "어떻게 물어볼 것인가(How to ask)"에 대한 답이다. 하나의 프롬프트를 어떻게 구성하면 더 좋은 결과가 나오는지에 집중한다.

그런데 실제 개발에서는 하나의 프롬프트로 끝나는 작업이 드물다. Tika 칸반 보드를 만든다고 생각해 보자. 데이터 모델을 설계하고, API를 구현하고, 프론트엔드를 만들고, 테스트를 작성하고, 배포 설정을 잡는다. 이 과정에서 AI는 수십 번의 추론을 반복하며, 매 턴마다 새로운 정보가 쌓인다. 파일을 읽고, 코드를 생성하고, 에러를 만나고, 수정한다.

이때 AI의 결과 품질을 결정하는 것은 마지막에 입력한 프롬프트 하나가 아니다. AI가 추론하는 시점에 "무엇을 보고 있는가"가 결정한다. 프로젝트 구조를 이해하고 있는가? 지금까지 내린 설계 결정을 기억하고 있는가? 관련 파일을 읽었는가? 아니면 3시간 전의 대화 잔해에 파묻혀 핵심을 놓치고 있는가?

이것이 컨텍스트 엔지니어링이 필요한 이유다.

---

## 컨텍스트 엔지니어링이란

Anthropic은 2025년 9월 공식 엔지니어링 블로그에서 컨텍스트 엔지니어링을 다음과 같이 정의했다.

> 컨텍스트(Context)란 LLM이 추론할 때 참조하는 토큰의 집합이다. 엔지니어링(Engineering)이란 그 토큰의 효용을 LLM의 고유한 제약 조건에 맞게 최적화하여 원하는 결과를 일관되게 달성하는 문제다.

쉽게 말하면, 프롬프트 엔지니어링이 "어떻게 물어볼 것인가"에 대한 것이라면, 컨텍스트 엔지니어링은 "AI가 추론하는 순간에 무엇을 보게 할 것인가"에 대한 것이다.

프롬프트 엔지니어링은 하나의 입력을 최적화한다. 컨텍스트 엔지니어링은 시스템 프롬프트, 도구 정의, 메시지 히스토리, 외부 데이터, 메모리 등 AI가 볼 수 있는 전체 환경을 설계한다.

### 프롬프트 엔지니어링 vs 컨텍스트 엔지니어링

| 구분 | 프롬프트 엔지니어링 | 컨텍스트 엔지니어링 |
|------|-------------------|-------------------|
| 초점 | 하나의 질문을 잘 구성하기 | AI가 보는 전체 환경을 설계하기 |
| 범위 | 단일 턴, 단일 프롬프트 | 다중 턴, 전체 세션 |
| 관리 대상 | 지시문, 예시, 제약 조건 | 시스템 프롬프트 + 도구 + 메모리 + 메시지 히스토리 + 외부 데이터 |
| 비유 | 좋은 질문을 하는 기술 | 전문가가 일할 수 있는 사무실 환경을 설계하는 기술 |
| 시점 | 요청 전에 한 번 | 매 추론마다 반복적으로 |

Shopify의 CEO 토비아스 뤼트케는 이를 이렇게 표현했다: "프롬프트 엔지니어링보다 컨텍스트 엔지니어링이라는 용어를 더 좋아한다. 이 용어가 핵심 역량을 더 잘 설명하기 때문이다 — LLM이 작업을 그럴듯하게 해결할 수 있도록 모든 컨텍스트를 제공하는 기술."

## 왜 컨텍스트가 유한한 자원인가

### 컨텍스트 윈도우의 구조

Claude Code의 컨텍스트 윈도우는 200K 토큰이다. 넉넉해 보이지만, 실제로 개발자가 쓸 수 있는 공간은 그보다 훨씬 작다. `/context` 명령어를 실행해 보면 컨텍스트가 어떻게 배분되어 있는지 확인할 수 있다.

```
┌─────────────────────────────────┐
│ 시스템 프롬프트                    │ ~15K 토큰
│ 시스템 도구 정의                   │ ~10K 토큰
│ MCP 도구 정의                     │ ~5K 토큰 (서버당)
│ CLAUDE.md (메모리 파일)            │ 가변
│ ─────────────────────────────── │
│ 대화 히스토리                      │ ← 여기가 계속 늘어남
│ (사용자 메시지 + AI 응답 +          │
│  도구 호출 결과 + 파일 읽기 결과)    │
│ ─────────────────────────────── │
│ 남은 공간                         │ ← 여기가 점점 줄어듦
└─────────────────────────────────┘
```

매 턴마다 대화 히스토리가 쌓이고, 남은 공간은 줄어든다. 문제는 단순히 공간이 부족해지는 것이 아니다.

### 컨텍스트 부패 (Context Rot)

Anthropic은 "컨텍스트 부패(Context Rot)"라는 현상을 소개했다. 컨텍스트 윈도우에 토큰이 많아질수록 모델이 해당 컨텍스트에서 정보를 정확하게 회수하는 능력이 떨어진다는 것이다.

이유는 트랜스포머 아키텍처의 자기 주의(Self-Attention) 메커니즘에 있다. n개의 토큰이 있으면 n² 쌍의 관계를 처리해야 한다. 토큰이 2배 되면 처리해야 할 관계는 4배가 된다. 컨텍스트가 길어질수록 모델의 "주의 예산(Attention Budget)"이 분산되어, 핵심 정보에 집중하는 능력이 약해진다.

이것이 의미하는 바는 명확하다. 컨텍스트에 더 많은 정보를 넣는다고 더 좋은 결과가 나오지 않는다. 관련 없는 정보는 관련 있는 정보를 가린다. 좋은 컨텍스트 엔지니어링이란, 원하는 결과를 이끌어내는 데 필요한 가장 작은 고신호(High-Signal) 토큰 집합을 찾는 것이다.

## Claude Code에서의 컨텍스트 엔지니어링 실전

이론을 이해했으니, Tika 칸반 보드 프로젝트에서 컨텍스트를 실제로 어떻게 관리하는지 살펴보자. Claude Code는 컨텍스트 엔지니어링을 위한 다양한 도구를 제공한다.

### 1. CLAUDE.md — 영구 컨텍스트의 설계

CLAUDE.md는 세션이 시작될 때마다 자동으로 컨텍스트에 로드되는 파일이다. 컨텍스트 엔지니어링 관점에서 보면, 이 파일은 "매번 AI가 반드시 보게 되는 영구 컨텍스트"다.

Anthropic의 표현대로, CLAUDE.md는 "사전 추론 시점에 무조건 투입되는 컨텍스트(naively dropped into context up front)"다. 그렇기 때문에 CLAUDE.md에 불필요한 내용이 많으면, 매 턴마다 주의 예산을 낭비하게 된다.

**나쁜 예: 모든 것을 담은 CLAUDE.md**

```markdown
# CLAUDE.md

## 프로젝트 소개
Tika는 칸반 보드 앱으로, Next.js 15를 사용하며...
(200줄의 프로젝트 설명)

## 기술 스택 상세
Next.js 15의 App Router는 기존 Pages Router와 달리...
(150줄의 기술 설명)

## 모든 API 엔드포인트 목록
GET /api/tickets - 전체 티켓 조회
POST /api/tickets - 티켓 생성
...
(100줄의 API 문서)

## 코딩 컨벤션 전체
변수명은 camelCase...
(80줄의 컨벤션 문서)
```

이 파일이 500줄이면, 매 턴마다 수천 토큰이 "프로젝트 소개"를 읽는 데 소비된다. 10번째 턴에서도, 50번째 턴에서도.

**좋은 예: 핵심만 담은 CLAUDE.md (200줄 이하)**

```markdown
# CLAUDE.md

## 프로젝트
Tika - 칸반 보드 TODO 앱 (Next.js 15 App Router, TypeScript)

## 기술 스택
- ORM: Drizzle (스키마: src/server/db/schema.ts)
- 검증: Zod (프론트/백엔드 공유)
- 스타일: Tailwind CSS 4
- DnD: @dnd-kit/core
- DB: Vercel Postgres (Neon)
- 테스트: Jest + React Testing Library

## 필수 규칙
- TDD: 테스트 먼저 작성 → 구현 → 리팩토링
- 요청한 파일만 수정. 다른 파일 수정 금지
- any 타입 사용 금지
- 테스트를 삭제하거나 비활성화하지 마
- API 응답: 성공 { data: ... }, 실패 { error: '메시지' }

## 상세 문서 위치
- API 설계: docs/API_SPEC.md
- 데이터 모델: docs/DATA_MODEL.md
- 컴포넌트 설계: docs/COMPONENT_SPEC.md
- 테스트 케이스: docs/TEST_CASES.md
```

핵심 차이가 보이는가? 상세 문서는 CLAUDE.md에 직접 쓰지 않고, 별도 파일의 경로만 기록한다. AI가 해당 정보가 필요할 때 직접 파일을 읽으면 된다. 이것이 Anthropic이 말하는 "경량 식별자(Lightweight Identifiers)" 전략이다 — 전체 데이터를 컨텍스트에 미리 넣지 않고, 경로나 참조만 유지하다가 필요할 때 동적으로 로드한다.

### 2. 적시 컨텍스트 (Just-in-Time Context)

Anthropic은 Claude Code의 컨텍스트 전략을 설명하면서 "적시(Just-in-Time)" 접근법을 강조했다. 관련 데이터를 미리 전부 로드하지 않고, 가벼운 식별자(파일 경로, 쿼리, 링크)만 유지하다가 런타임에 도구를 사용하여 필요한 데이터를 동적으로 로드하는 방식이다.

Claude Code에서 이 전략은 `glob`과 `grep` 같은 도구로 구현된다. AI가 파일 시스템을 탐색하면서 필요한 파일만 선택적으로 읽는다. 스탈 인덱싱이나 복잡한 구문 트리 문제를 우회할 수 있다.

**실전 예시: Tika 프로젝트에서의 적시 컨텍스트**

```
티켓 완료 시 completedAt이 설정되는 로직에 버그가 있어.
Done으로 이동했는데 completedAt이 null이야.

관련 파일을 찾아서 원인을 분석해줘.
```

이 프롬프트를 받은 Claude Code는 다음과 같은 흐름으로 동작한다:

1. `grep -r "completedAt" src/` — 관련 파일 탐색
2. `app/api/tickets/[id]/complete/route.ts` 발견 → 해당 파일만 읽기
3. `src/server/db/schema.ts` — completedAt 컬럼 정의 확인
4. 원인 분석 후 수정 제안

전체 프로젝트를 미리 컨텍스트에 넣은 것이 아니다. "completedAt"이라는 키워드를 단서로 관련 파일만 선택적으로 읽었다. 이것이 적시 컨텍스트다.

반면 아래처럼 하면 컨텍스트를 낭비한다:

```
# 나쁜 예: 모든 파일을 미리 읽히기
src/ 디렉토리 전체를 읽고, 
티켓 완료 시 completedAt이 설정되는 로직의 버그를 찾아줘.
```

이렇게 하면 관련 없는 파일 수십 개가 컨텍스트에 올라와서 주의 예산을 낭비한다.

### 3. 컨텍스트 압축 (Compaction)

대화가 길어지면 컨텍스트 윈도우가 차오른다. 이때 Claude Code의 `/compact` 명령어를 사용한다.

Anthropic에 따르면, 압축은 "대화의 내용을 고충실도(High-Fidelity)로 요약하여 성능 저하를 최소화하면서 에이전트가 작업을 계속할 수 있게 하는" 기법이다. Claude Code는 메시지 히스토리를 모델에 전달하여 요약하고, 아키텍처 결정, 미해결 버그, 구현 세부사항은 보존하면서 중복된 도구 출력이나 메시지는 버린다.

**수동 압축과 자동 압축의 차이**

Claude Code는 자동 압축(auto-compaction)을 지원하지만, 마일스톤마다 수동으로 실행하는 것이 보존할 내용을 제어할 수 있어서 더 효과적이다.

```bash
# 기본 압축
/compact

# 보존할 내용을 명시한 압축
/compact 아키텍처 결정사항, 변경된 파일 목록, 
        미해결 TODO 항목은 반드시 보존할 것.
        탐색 과정의 시행착오 대화는 제거.
```

**타이밍 가이드:**

| 시점 | 행동 |
|------|------|
| 컨텍스트 사용률 60~70% | 압축을 고려하기 시작 |
| 하나의 기능 구현 완료 후 | `/compact` + `git commit` |
| 작업 주제가 완전히 바뀔 때 | `/clear`로 완전 초기화 |
| 긴 디버깅 세션 후 | `/compact`로 결론만 보존 |

### 4. 구조화된 메모 (Structured Note-Taking)

Anthropic은 "구조화된 메모(Structured Note-Taking)" 또는 "에이전틱 메모리(Agentic Memory)"라는 기법을 소개했다. 에이전트가 정기적으로 컨텍스트 윈도우 외부에 메모를 작성하고, 나중에 필요할 때 다시 컨텍스트로 가져오는 방식이다.

Claude Code에서는 이를 CLAUDE.md의 `#` 단축키나 `/memory` 명령어로 실현할 수 있다.

**실전 예시: 마일스톤별 메모 전략**

Tika 프로젝트의 백엔드 구현 단계를 예로 들어 보자.

```
[1단계: 스키마 정의 완료]
→ 구현 완료, 테스트 통과
→ /memory로 결정사항 기록:
  "position은 정수형, 1024 간격. 
   간격이 1 이하면 칼럼 전체 재정렬."
→ git commit
→ /compact

[2단계: GET /api/tickets 구현]
→ 구현 완료, 테스트 통과
→ /memory로 기록:
  "Done 칼럼은 completedAt 기준 24시간 필터 적용.
   서버에서 필터링하여 응답."
→ git commit
→ /compact

[3단계: POST /api/tickets 구현]
→ 이전 단계의 결정사항은 CLAUDE.md에 있으므로
  /compact 후에도 AI가 기억하고 있음
```

각 마일스톤에서 핵심 결정사항을 메모리에 기록하고, 컨텍스트를 압축한다. 압축 후에도 CLAUDE.md에 기록된 내용은 다음 턴에 자동으로 로드되므로, AI는 이전 단계의 결정사항을 "기억"한다.

### 5. 도구 설계 — MCP의 컨텍스트 영향

MCP(Model Context Protocol) 서버는 AI에게 외부 도구를 제공한다. 그런데 각 MCP 도구의 정의(이름, 설명, 파라미터 스키마)는 컨텍스트 윈도우에 상주한다. MCP 서버를 5개 연결하고 각각 10개의 도구를 제공하면, 도구 정의만으로 수천 토큰을 차지한다.

Anthropic은 이를 경고하며, "가장 흔한 실패 모드 중 하나는 너무 많은 기능을 커버하거나, 어떤 도구를 사용해야 할지 모호한 결정 지점을 만드는 비대한 도구 셋"이라고 했다.

**도구 셋 관리 원칙:**

```bash
# 현재 연결된 MCP 서버 확인
/mcp

# 이번 세션에 필요한 것만 유지
# 예: Tika 백엔드 구현 세션이라면
#   ✅ PostgreSQL MCP — DB 직접 확인용
#   ✅ GitHub MCP — 이슈/PR 관리
#   ❌ Sentry MCP — 아직 배포 전이므로 불필요
#   ❌ Slack MCP — 코딩 작업과 무관
```

도구가 많을수록 AI의 판단이 흐려진다. "이 상황에서 어떤 도구를 써야 하는가"를 사람도 명확히 답할 수 없다면, AI는 더더욱 못한다. 사용하지 않는 MCP 서버는 비활성화하라.

### 6. .claudeignore — 불필요한 파일 차단

`.claudeignore` 파일은 Claude Code가 파일 시스템을 탐색할 때 무시할 경로를 지정한다. 이 파일이 중요한 이유는, Claude Code가 프로젝트 구조를 파악하기 위해 디렉토리를 스캔할 때 불필요한 파일까지 읽지 않도록 방지하기 때문이다.

```
# .claudeignore
node_modules/
.next/
dist/
coverage/
*.lock
.git/
drizzle/meta/
```

`node_modules/`만 제외해도 수만 개의 파일이 탐색 대상에서 빠진다. 한 실험 보고에 따르면 `.claudeignore` 설정만으로 토큰 소비를 가장 크게 줄일 수 있었다고 한다.

## 컨텍스트 엔지니어링 워크플로우

지금까지 다룬 기법들을 하나의 워크플로우로 정리하면 다음과 같다.

### 세션 시작 시

```bash
# 1. 컨텍스트 상태 확인
/context              # 현재 컨텍스트 배분 확인
/cost                 # 비용 기준점 확인

# 2. 불필요한 MCP 서버 비활성화
/mcp                  # 연결된 서버 확인, 불필요한 것 제거

# 3. 작업 범위 명시적으로 전달
"이번 세션에서는 POST /api/tickets 엔드포인트만 구현한다.
 관련 파일: app/api/tickets/route.ts, src/server/db/schema.ts
 참고 문서: docs/API_SPEC.md의 POST /api/tickets 섹션"
```

### 작업 중

```bash
# 컨텍스트 사용률 모니터링
/context              # 주기적으로 확인

# 마일스톤마다 기록 + 압축
/memory               # 핵심 결정사항 기록
/compact              # 시행착오 대화 제거

# 비용 확인
/cost                 # 예상보다 높으면 /compact 고려
```

### 작업 전환 시

```bash
# 완전히 다른 작업으로 넘어갈 때
/memory               # 현재 작업의 결론 기록
git commit            # 변경사항 커밋
/clear                # 컨텍스트 완전 초기화

# 유사한 작업을 이어갈 때
/compact 아키텍처 결정사항과 완료된 작업 목록만 보존
```

## 요약: 프롬프트 엔지니어링에서 컨텍스트 엔지니어링으로

| 원칙 | 실천 |
|------|------|
| 컨텍스트는 유한한 자원이다 | `/context`로 사용량 모니터링 |
| 적을수록 좋다 | CLAUDE.md는 200줄 이하, 상세 문서는 별도 파일 |
| 적시에 로드한다 | 전체 프로젝트를 미리 읽히지 말고, 필요한 파일만 |
| 정기적으로 정리한다 | 마일스톤마다 `/compact`, 작업 전환 시 `/clear` |
| 메모로 연결한다 | `/memory`로 핵심 결정사항을 세션 너머로 전달 |
| 도구를 최소화한다 | 불필요한 MCP 서버 비활성화, `.claudeignore` 설정 |

프롬프트 엔지니어링이 "좋은 질문을 하는 기술"이라면, 컨텍스트 엔지니어링은 "전문가가 최고의 성과를 낼 수 있는 환경을 설계하는 기술"이다. 아무리 좋은 질문을 해도 전문가의 책상 위가 관련 없는 서류로 뒤덮여 있다면 좋은 결과를 기대하기 어렵다. AI도 마찬가지다.

이 책에서 다루는 설계 문서 작성(4장), TDD 사이클(3.6절), 단계적 작업 분해(3.2절) 모두 결국 하나의 목표로 수렴한다: AI가 추론하는 순간에 올바른 컨텍스트를 제공하는 것. 컨텍스트 엔지니어링은 이 모든 실천을 관통하는 메타 원칙이다.
